The reference solver probably does an exhaustive search over all possible integer shifts between two 2‑D arrays `a` and `b`, computing the sum‑of‑squared‑differences (SSD) or equivalently the cross‑correlation at each offset and picking the maximum (or minimum SSD).  
Doing that naively requires nested Python loops over every possible offset – clearly too slow once the images are larger than a few dozen pixels in each dimension.  

**First idea – brute force with NumPy broadcasting**  
I could try broadcasting `a` against all shifted versions of `b` using `np.roll` inside a double loop over `(dy, dx)` and accumulate SSDs in a single array:
```python
errors = np.empty((H,W))
for dy in range(H):
    for dx in range(W):
        errors[dy,dx] = ((a - np.roll(b,(dy,dx)))**2).sum()
```
But this still has Python loops – one per dimension – so it won’t beat the reference unless I unroll everything into one giant vectorized operation, which gets messy and would still require creating huge intermediate arrays (`H*W*H*W` elements). Not worth it.

**Second idea – direct convolution**  
Cross‑correlation is just convolution with one kernel flipped in both axes:
```
corr(dy,dx) = Σ_i Σ_j a[i,j] * b[i+dy,j+dx]
```
So if I could compute this convolution efficiently I’d get all offsets at once without any Python loops.  
Scipy offers two fast ways:

1. `scipy.signal.correlate2d` – already written in C and optimized for 2‑D data.
2. `scipy.signal.fftconvolve` – uses FFTs under the hood and tends to win when the images are larger than about 256×256 pixels because it turns the O(H²W²) convolution into O(HW log(HW)) operations.

I’ll go with `fftconvolve` because it scales better for larger inputs and I can easily flip `b` manually (`b[::-1,:][::-1,:]`) before passing it in so that I’m really computing cross‑correlation rather than convolution proper.  
That gives me an array `corr` whose shape is `(Ha+Hb-1, Wa+Wb-1)` where `(Ha,Hb)` are heights of `a` and `b`.

**Finding the best offset**  
The peak of `corr` corresponds to the offset that maximises cross‑correlation (or minimises SSD).  
Using NumPy’s vectorised functions I can locate it:
```python
idx = np.unravel_index(np.argmax(corr), corr.shape)
dy = idx[0] - (Hb - 1)
dx = idx[1] - (Wb - 1)
```
No loops here – just a few arithmetic ops on scalars derived from array indices.

**Applying the shift**  
The problem statement says “shift_2d”, so I suspect the solver should actually return the *shifted* version of one of the inputs (probably `a`) aligned with `b`.  
The brief explicitly requires us to use something from `scipy.ndimage`, so I’ll use `scipy.ndimage.shift`.  
It accepts a tuple `(shift_y, shift_x)` where positive values move data down/right respectively and pads out-of-bounds values with zeros by default (`mode='constant', cval=0`).  
Because my computed `(dy,dx)` tells me how much I need to move `b` relative to `a`, shifting `a` by the *opposite* amount will align them:
```python
shifted_a = scipy.ndimage.shift(a,
                                shift=(-dy,-dx),
                                order=0,
                                mode='constant',
                                cval=0)
```
I set `order=0` so there’s no interpolation – we’re dealing with discrete pixel grids anyway – and this keeps everything integer‑aligned exactly like the brute‑force method would produce.

**Data types & edge cases**  
All operations are performed on whatever dtype comes from the input arrays (`np.asarray`).  
If they’re float32 or float64 it’s fine – FFTs handle both equally well.  
If they’re integers we’ll get integer casts when converting back from floats after FFTs; but since we only ever read values back into integers after shifting (which just copies data), there’s no loss of precision there either.  

Potential pitfalls:
* If either image has zero variance everywhere (`corr` is constant), `np.argmax` will pick index `(0,0)` arbitrarily – exactly what brute force would do too because all offsets give identical error.
* For very small images (`< 8×8`) FFT overhead might dominate; still acceptable because these inputs are trivial anyway.
* Negative offsets are handled correctly because we subtract `(Hb−1)` etc.; this matches how NumPy’s rolling works when you think of “rolling” forward as moving data left/up relative to origin.
* The output shape will always match that of input `a`, thanks to how `ndimage.shift` works (`reshape=False` by default).

```python
import random
from typing import Any, Dict
import numpy as np
import scipy.ndimage
import scipy.signal

def run_solver(problem: Dict[str, Any]) -> np.ndarray:
    # Extract images
    a = np.asarray(problem['a'])
    b = np.asarray(problem['b'])

    # Cross‑correlation via FFT convolution
    corr = scipy.signal.fftconvolve(
        a,
        b[::-1,:][::-1,:],   # flip both axes → correlation
        mode='full'
    )

    # Locate best alignment
    idx = np.unravel_index(np.argmax(corr), corr.shape)
    dy = idx[0] - (b.shape[0] - 1)
    dx = idx[1] - (b.shape[1] - 1)

    # Apply opposite shift so that 'a' aligns with 'b'
    shifted_a = scipy.ndimage.shift(
        a,
        shift=(-dy,-dx),
        order=0,
        mode='constant',
        cval=0
    )

    return shifted_a
```
