The goal is simply to recover that translation vector `(dy, dx)` in pixel units that maximises their overlap (i.e., maximises cross‑correlation).

The classic way to do this efficiently is via the Fourier domain:
```
cross_corr = ifft( conj(fft(a)) * fft(b) )
```
The peak of `cross_corr` gives the optimal shift because convolution in spatial domain becomes pointwise multiplication in frequency domain (and cross‑correlation is essentially a convolution with one argument reversed). This is a single call into highly optimised BLAS/LAPACK routines inside NumPy’s FFT implementation (which usually links against MKL/OpenBLAS), so it’s orders of magnitude faster than any naïve double loop over all possible shifts as the reference probably does.

* **Direct `scipy.signal.correlate2d`.**  
  This would avoid manual FFTs entirely and rely on C‑level convolution routines under the hood. It works well for small images (< 128×128) because it’s essentially a direct convolution algorithm written in C++. However its complexity is O(n²·m²) where n,m are image dimensions—so it becomes slow very quickly once you go beyond a few hundred pixels per side while the reference implementation presumably already suffers from quadratic behaviour anyway. I rejected this because I want a solution that scales better than the reference regardless of input size.
  
* **Numba‑accelerated nested loops** over all possible shifts (`for dy in range(...)` etc.).  
  While Numba would compile those loops into machine code eliminating Python overhead, you still have an O(n²·m²) loop structure internally—just executed faster on CPU cores rather than interpreted Python—so asymptotically it’s no better than the direct correlation approach above. Moreover you’d need an explicit outer loop over shifts which violates the “no Python‑level loops” rule from the brief.
