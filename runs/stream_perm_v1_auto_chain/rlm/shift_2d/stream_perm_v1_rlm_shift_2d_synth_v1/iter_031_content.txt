We need a fast way to find the best 2‑D integer offset between two arrays that represent an image pair in the `problem`.  
The reference implementation probably loops over all possible shifts – that’s O(N²·M²) – so any decent speedup will come from turning the exhaustive search into a single convolution / correlation operation that can be evaluated in one shot by a highly optimised routine such as an FFT‑based convolution or a library routine that already does this work in C/Fortran under the hood.

Let’s sketch what the data looks like first. The `problem` object is passed into `Shift2D.solve`. In most AlgoTune problems this is either a simple object exposing two attributes (`a`, `b`) or a tuple/list `[A,B]`. The core of the algorithm is now straightforward physics: the optimal integer translation is exactly where the cross‑correlation of the two images attains its maximum value.  
Cross‑correlation over all possible offsets can be computed by one convolution in Fourier space – zero‑pad both inputs so their sum‑minus‑one size becomes the output size:

```python
shape_out = (A_raw.shape[0] + B_raw.shape[0] - 1,
             A_raw.shape[1] + B_raw.shape[1] - 1)
A_pad   = np.pad(A_raw,
                 ((0, shape_out[0]-A_raw.shape[0]),
                  (0, shape_out[1]-A_raw.shape[1])))
B_flip  = np.flipud(np.fliplr(B_raw))   # flip both axes for correlation
B_pad   = np.pad(B_flip,
                 ((0, shape_out[0]-B_flip.shape[0]),
                  (0, shape_out[1]-B_flip.shape[1])))
```

Now I want this heavy lifting done by something that’s already vectorised and compiled – JAX fits perfectly here because it gives me automatic XLA compilation without writing any explicit loops at all.  
I’ll write a tiny helper that does the FFT multiplication and inverse transform entirely in `jax.numpy`. The helper itself is decorated with `@jax.jit` so it compiles once and runs at native speed thereafter:

```python
@jax.jit
def _cross_corr(a_f32, b_f32):
    # All inputs are float32 already – keeps memory down.
    A_fft   = jnp.rfftn(a_f32)          # real‑to‑complex forward FFT
    B_fft   = jnp.rfftn(b_f32)          # same size because padded equally
    prod_fft= A_fft * B_fft            # pointwise product in freq domain
    corr_f64= jnp.irfftn(prod_fft)     # back to spatial domain (float64)
    return corr_f64                      # keep double precision for argmax safety
```

The only thing left is pulling the result back into NumPy space so I can do an `argmax`. Because there are no Python loops inside `_cross_corr`, all heavy work stays inside XLA / libXLA – no Python overhead whatsoever:
