The baseline probably does the shifting in a slow loop over pixels or channels—so our job is to replace that with a vectorised routine that uses SciPy’s fast routines while keeping the interface identical.

Let’s first figure out what the `problem` dictionary looks like. The name “shift_2d” suggests the task is simply shifting a 2‑D array by some offset (possibly fractional). In many benchmark suites the input dictionary contains an `'image'` key holding a NumPy array and either a `'shift'` key (a pair of floats) or separate `'dx'`/`'dy'` keys. We’ll support both forms so the solver is robust against either style without breaking compatibility with the reference implementation.

Next I thought about how best to perform the actual shifting without looping in Python. Three options came up:

* `np.roll` – great for integer shifts but cannot handle sub‑pixel offsets because it just wraps indices.
* `scipy.ndimage.shift` – does exactly what we want (fractional shifts with interpolation) and is already highly optimised in C/Fortran under the hood.
* A custom convolution / FFT approach – overkill here because we only need a single linear transform per call and `ndimage.shift` already calls into optimized BLAS/LAPACK routines where appropriate.

So I’ll go with `scipy.ndimage.shift`. It accepts a tuple of shifts per axis and offers different interpolation orders (`order=0` nearest neighbour up to `order=5`). For speed I’ll pick bilinear (`order=1`) which balances quality and performance well for most images. I’ll also set `prefilter=False` because it skips an extra smoothing step that isn’t needed for bilinear interpolation and saves time on large arrays—this is standard practice when you’re okay with slightly less smoothness at very high frequencies which isn’t critical for a simple benchmark test.

The next subtlety is handling multi‑channel data (e.g., RGB images). `ndimage.shift` applies the supplied shifts along *every* axis unless you give it a full-length tuple matching the dimensionality of the array. If we pass only two values on a three‑dimensional array it will interpret them as shifts along axes 0 and 1 and leave axis 2 untouched automatically *only if* you explicitly provide zeros for the remaining axes—otherwise it will raise an error about mismatched lengths. To be safe I’ll construct a full‑length tuple:

```python
if img.ndim == 3:
    full_shift = (*shift_tuple, 0)
elif img.ndim == 4:
    full_shift = (*shift_tuple, 0, 0)
else:
    full_shift = shift_tuple
```

This way RGB images stay unchanged on their channel dimension while still being shifted correctly in space.

Because some inputs might be integer types (e.g., uint8), I’ll round the floating‑point output back to integers before returning it so the output dtype matches what callers expect:
