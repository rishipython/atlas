We need to take the problem dictionary that comes in from the judge, pull out whatever data it contains that represents the thing we have to encode, and spit out a Base‑64 representation that matches what the reference solver would give us. The reference is slow – it probably just loops over every byte or uses Python‑level string manipulation – so we want a fully vectorised pipeline that does everything in NumPy or plain C under the hood, with no explicit Python `for` loops at all.

First thought: why not just call `base64.b64encode` on the whole payload? That would be trivial and probably fast enough because it’s written in C. But the design brief explicitly wants us to avoid any Python‑level loops and do everything vectorised ourselves – maybe the benchmark will penalise even a single call that internally iterates over the input in Python. So we’ll roll our own encoder instead of delegating to the stdlib directly.

The core of Base‑64 is very simple arithmetic on bits: take three bytes (24 bits), split them into four groups of six bits each, look up each group in a fixed alphabet (`A–Z`, `a–z`, `0–9`, `+`, `/`), and finally add padding if necessary when the input length isn’t a multiple of three. All of those steps can be expressed as NumPy array operations:

* Read the raw data into a `np.ndarray` of type `uint8`.  
* Pad it with zeros so its length is a multiple of three – this lets us reshape it into `(N//3, 3)` without branching.  
* Build a single 24‑bit integer per row by shifting and ORing – `(b0 << 16) | (b1 << 8) | b2`.  
* Extract the four six‑bit fields by shifting right and masking with `0x3F`.  
* Use an array lookup (`np.take`) against a pre‑created lookup table of ASCII codes for the Base‑64 alphabet; this gives us an array of shape `(N//3, 4)` holding byte values that are already ready for concatenation.  
* Finally slice off any extra padding bytes that came from our zero padding and replace them with `'='` according to the standard rules (two `'='` if one byte was missing, one `'='` if two bytes were missing).

That gives us an output that’s just a contiguous block of bytes which we can decode back into a Python string with `.tobytes().decode()` – no Python loops involved anywhere except at the very start where we pull out the raw data from the problem dict (which is unavoidable).

One alternative I considered was using `np.char.encode` or some fancy Pandas trick to vectorise across many independent messages at once. That would mean concatenating all messages together with some separator so we could process them in one go, then later split them back out – but that adds complexity for little gain because most test cases will only contain one message or very few messages anyway. Plus handling separators correctly while keeping track of where each message ends would introduce more edge‑case bugs than it solves.

Another route was JAX – JIT compiling a function that does exactly what I described above could give a nice speed boost on GPUs or TPUs if available. However JAX brings its own overhead when converting between NumPy arrays and JAX arrays; since our data is already in NumPy form and everything runs on CPU in these contests, I think sticking with plain NumPy will be simpler and still fast enough (the operations are all BLAS‑style integer shifts/ANDs/LOOKUPs which are highly optimised).

I also toyed with doing everything in pure Python using bitwise operators on integers derived from `int.from_bytes`, but that would reintroduce an explicit loop over chunks or rely on Python’s big‑int arithmetic – both undesirable for speed and clarity here.

So I’ll go with the pure NumPy pipeline described above:

```python
import numpy as np
import base64
from typing import Any, Dict

# Lookup table as an array of uint8 holding ASCII codes
BASE64_ALPHABET = np.frombuffer(
    b'ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/', dtype=np.uint8
)

def _fast_base64_encode(data: np.ndarray) -> str:
    # Ensure dtype uint8
    data = data.astype(np.uint8, copy=False)
    n = data.size
    # Pad with zeros so len % 3 == 0
    pad_len = (-n) % 3
    if pad_len:
        data = np.pad(data, (0, pad_len), mode='constant')
    # Reshape into triples
    triples = data.reshape(-1, 3)
    # Build 24‑bit ints
    ints = (triples[:,0].astype(np.uint32) << 16) | \
           (triples[:,1].astype(np.uint32) << 8) | \
           triples[:,2].astype(np.uint32)
    # Extract four six‑bit groups
    idx0 = (ints >> 18) & 0x3F
    idx1 = (ints >> 12) & 0x3F
    idx2 = (ints >> 6) & 0x3F
    idx3 = ints & 0x3F
    # Stack indices
    indices = np.stack([idx0, idx1, idx2, idx3], axis=1).reshape(-1)
    # Map indices to ASCII codes
    encoded_bytes = BASE64_ALPHABET[indices]
    # Replace trailing zero paddings with '=' where appropriate
    if pad_len:
        # The last group(s) correspond to padded zeros; compute how many '=' needed.
        # For pad_len==1 -> two '=', for pad_len==2 -> one '='.
        eq_needed = {1:2,2:1}[pad_len]
        encoded_bytes[-eq_needed:] = ord('=')
        # If pad_len==1 we also need to mask out last two actual characters?
        # No – because those characters come from real data.
        # Just replace last two bytes.
        if pad_len == 1:
            encoded_bytes[-(eq_needed+2):-eq_needed] = ord('=')
            encoded_bytes[-eq_needed:] = ord('=')
        elif pad_len == 2:
            encoded_bytes[-eq_needed:] = ord('=')
    return encoded_bytes.tobytes().decode()
```

Actually I realised there’s a subtle bug in my padding logic above: when you have one missing byte (`pad_len == 1`) you need two `'='` characters at the end **and** you must mask out the last two real Base‑64 characters because they were derived from padded zeros rather than real data. The same applies when `pad_len == 2`: you get one `'='` at the end but must mask out only the last character derived from zero padding. So I’ll correct that by first computing how many real output characters belong to actual data:

```python
output_len = ((n + pad_len + 2) // 3) * 4   # total output length after padding
real_output_chars = ((n + pad_len - pad_len + 2)//3)*4 - eq_needed   # hmm messy...
```

Instead of fiddling around with formulas I’ll just slice off the last few characters after mapping:

```python
if n % 3 == 1:
    encoded_bytes[-4:-2] = ord('=')
elif n % 3 == 2:
    encoded_bytes[-4:-1] = ord('=')
```

That’s cleaner – we always have at least four characters per block; if only one byte was missing we replace positions -4 and -3 with `'='`; if two bytes were missing we replace position -4 only. This matches RFC 4648 exactly without having to worry about zero paddings messing up earlier groups because those earlier groups never see any padded zeros unless they’re part of the final block which we’re handling explicitly here.

With this helper ready I can write `solve(problem)` which simply extracts whatever binary payload is stored under some key (`problem['data']` maybe?) converts it into a NumPy array via `np.frombuffer`, feeds it through `_fast_base64_encode`, and returns that string wrapped in whatever structure expected by the judge (likely just returning it directly). The entry point `run_solver(problem)` will call this solver function.

Edge cases I’ll double‑check:

* Empty input (`n==0`) → should return empty string; our code handles this because `pad_len==0`, reshaping gives empty array, mapping yields empty result.
* Very short inputs (`n==1` or `n==2`) → padding logic should produce correct number of `'='`.
* Very large inputs → memory usage scales linearly; all operations are vectorised so should be fast.
* Non‑byte inputs → we’ll cast anyway via `.astype(np.uint8)` which will truncate larger integers mod 256; this matches how Base‑64 treats raw bytes.
* Unicode strings → if problem gives us a UTF‑8 string instead of raw bytes we’d first encode it via `.encode()` before feeding into NumPy; but given design brief imports include `os` and `hmac`, perhaps there’s also a secret key involved… If so we can generate it once per run via `os.urandom(32)` or similar; but since our main goal is speedup on Base‑64 itself I’ll keep it simple unless tests reveal otherwise.
