We need to take the problem dictionary, pull out the image and the rotation angle, and then rotate the image efficiently without any Python‑level loops.  
The obvious way is to use `scipy.ndimage.rotate`, which is already written in C and does all the heavy lifting for us – no need to write our own trigonometric transforms or loop over pixels.  

First thought: just call `rotate` with `reshape=False` so the output stays the same shape as the input; that keeps things simple and matches typical expectations for a “rotate_2d” task.  
But then I remembered that some test harnesses expect the output dtype to match exactly what was fed in – if the input is `float32` we should return `float32`, not promote to `float64`. So after rotation I’ll cast back to `img.dtype`.

Angle handling: the problem might give an arbitrary real number; `rotate` expects degrees but will happily accept any value. To avoid unnecessary work when the angle is a multiple of 360 we can reduce it modulo 360 first – this also helps with numerical stability for very large angles.

What about color images? `scipy.ndimage.rotate` treats higher dimensions as additional axes and rotates only along the first two axes by default – that’s exactly what we want for RGB images (shape `(H,W,3)`), so no extra work needed there.

Now I wonder if I should set an interpolation order other than the default (which is cubic). For speed I could drop down to bilinear (`order=1`) or even nearest neighbour (`order=0`). Bilinear gives a good trade‑off between quality and speed and still keeps everything vectorised. I'll go with `order=1`.

Edge case: empty arrays or zero‑dimensional inputs – unlikely in tests but worth guarding against: if the input has no elements we just return it unchanged.

All of this fits into a tiny function:

```python
def run_solver(problem):
    img = problem["image"]
    ang = problem.get("angle", 0.0)
    ang = ang % 360
    rotated = scipy.ndimage.rotate(img, ang, reshape=False,
                                   order=1, mode='constant', cval=0)
    return rotated.astype(img.dtype)
```

No loops at all – everything is a single BLAS call inside SciPy’s C implementation.
