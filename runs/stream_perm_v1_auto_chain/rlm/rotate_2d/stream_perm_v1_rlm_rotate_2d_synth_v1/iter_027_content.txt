The task is essentially a single‑shot image rotation – given an array and an angle in degrees – return the rotated image without changing its size (so `reshape=False`).  

The baseline probably builds a transformation matrix and applies it pixel‑by‑pixel in pure Python loops. That’s slow because every pixel requires trigonometric calculations and interpolation in Python land.  
SciPy already ships an efficient `ndimage.rotate` routine written in C that does exactly what we want: it handles multi‑channel images automatically (each channel is rotated independently), it supports different interpolation orders (`order`), fill modes (`mode`), and constant values (`cval`).  
So the plan is:

1. **Parse the input dictionary** – pull out the mandatory `'data'` key (the image) and optional parameters `'angle'`, `'order'`, `'mode'`, `'cval'`.  
   *If any are missing use sensible defaults:* `angle=0`, `order=1` (linear), `mode='constant'`, `cval=0`.  
   We’ll cast everything explicitly so type conversions are clear (`float(angle)`, `int(order)`, etc.).  

2. **Early exit** – if the normalized angle (`%360`) is zero just return a copy of the input array (`img.copy()`). This avoids calling the heavy routine when unnecessary and keeps behaviour identical to the reference (which probably also does this).  

3. **Call `scipy.ndimage.rotate`** – pass `reshape=False` so the output keeps the same dimensions as the input (matching most image‑processing conventions).
