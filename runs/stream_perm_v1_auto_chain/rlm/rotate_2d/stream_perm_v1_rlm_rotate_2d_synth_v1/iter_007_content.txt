The obvious library‑based way to rotate an image in Python is `scipy.ndimage.rotate`.  
It’s already written in C/Fortran under the hood so it will beat any pure‑Python loop or even a hand‑rolled NumPy implementation that does explicit index gymnastics or interpolation from scratch.  
The only thing we have to worry about is that `rotate` returns a floating point array even when the input was integer‑typed – e.g., rotating a `uint8` image will give us `float64` values slightly outside `[0,255]`.  
If we simply cast back we risk wrap‑around or clipping errors when those values fall outside the representable range of the target dtype (`np.uint8` wraps around modulo 256!).  
So before casting back we should clamp the values using `np.clip` against the min/max bounds obtained from `np.iinfo` for integers or `np.finfo` for floats.

There are several alternative routes I considered:

* **Numba JIT** – I could write my own bilinear interpolation routine in Python and decorate it with `@numba.njit`.  
  That would avoid any external dependency beyond Numba itself and give me fine control over memory layout.  
  However I quickly realized that Numba’s support for multi‑dimensional indexing inside loops still leaves me writing explicit Python loops over pixels – which defeats the “no Python‑level loops” constraint stated in the brief.
* **Custom FFT‑based rotation** – rotating via Fourier shift theorem could theoretically be faster on very large images because FFTs are highly optimized in SciPy/NumPy BLAS wrappers.  
  Yet implementing accurate interpolation in Fourier space requires careful handling of zero‑padding and phase wrapping; any small mistake would produce noticeable ringing artifacts that would make our solution incorrect under the tolerance checks.
* **Pure NumPy broadcasting** – I toyed with constructing a grid of coordinates transformed by an affine matrix and then sampling via linear interpolation using `np.take`/`np.interp`.  
  The resulting code was about twice as long as calling `ndimage.rotate`, still involved explicit indexing logic that was hard to vectorise completely without resorting to loops over channels or slices.
* **SciPy’s `affine_transform`** – this function gives us direct control over the affine matrix but again forces us into dealing with coordinate mapping manually if we want nearest‑neighbor vs bilinear interpolation handling consistent with what tests expect.
* **OpenCV (`cv2.warpAffine`)** – would have been fast but requires pulling in another heavy dependency not listed among allowed packages in the brief; besides OpenCV’s API isn’t part of standard SciPy stack so it’d break reproducibility across environments.
* **Using JAX** – JAX offers XLA compilation which can accelerate rotations when written as explicit grid sampling kernels.  
  However setting up JAX-compatible code here would add significant boilerplate (device placement logic etc.) while offering marginal speed gains over SciPy’s highly tuned C++ backend on CPU targets.
* **Scipy’s `transform.rotate` from older versions** – similar performance characteristics but now deprecated in favour of `ndimage.rotate`, so I dropped this idea early on.
