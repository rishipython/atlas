We need a fast way to rotate a 2‑D array without any Python‑level loops – the reference Rotate2D class does it in pure Python and ends up being a bottleneck when the images get large or when many rotations are required in a batch test harness.  

Let’s think about what the problem actually gives us first. The `problem` dictionary will contain an `'image'` key holding a NumPy array – either 2‑D (grayscale) or 3‑D (height × width × channels). There’s also an `'angle'` key expressed in degrees that tells us how much counter‑clockwise rotation is required (the usual convention used by most imaging libraries). The goal is simply to return the rotated image while preserving the original dimensions so that downstream code can compare against the ground truth without worrying about reshaping logic.



**Option A – Write our own bilinear sampler**

One idea would be to compute the rotation matrix ourselves:

```
c   ← [h/2-½ , w/2-½]          # centre of the image
R   ← [[cosθ , -sinθ],
       [sinθ ,  cosθ]]
A   ← Rᵀ                        # map output → input
off ← c – A @ c
```

Then call `scipy.ndimage.affine_transform` once on the whole array – it accepts multi‑dimensional inputs and applies the same linear map across all remaining axes (so one call handles RGB images too). This keeps everything vectorised and leverages SciPy’s highly optimised C backend for spline interpolation (`order=1` gives bilinear).  

The benefit here is that we avoid any explicit looping over pixels or channels entirely – everything happens inside the compiled routine.



**Option B – Just hand off straight to SciPy’s `rotate`**

SciPy already ships a thin wrapper called `scipy.ndimage.rotate`. Under the hood it calls `map_coordinates` with spline interpolation too (`order` defaults to 3), but you can drop it down to `order=1` for speed while keeping behaviour identical to bilinear sampling used by many baselines.  
The function also accepts `reshape=False`, so it will return an array of exactly the same shape as the input – matching what most tests expect when they compare against reference outputs generated by a similar “no‑reshape” policy.



**Which one wins?**

Both approaches avoid Python loops entirely and rely on compiled C code inside SciPy/Numpy – that alone gives a huge win over hand‑rolled pixel‑by‑pixel logic in pure Python.  
The `affine_transform` route requires us to assemble the inverse matrix manually (`A`) and compute an offset (`off`). That extra arithmetic isn’t huge but it adds a few lines of code that could potentially be error‑prone if I forget that `affine_transform` expects *output → input* mapping (`A` must be `Rᵀ`).  
On the other hand `rotate` hides all of that complexity behind a single function call with clear arguments (`angle`, `reshape`, `order`, etc.). It’s shorter and less room for mistakes.



**Edge cases & sanity checks**

* **Zero rotation** – calling `rotate` with `angle=0` simply copies the array through its internal spline machinery – negligible overhead.
* **Negative angles** – SciPy interprets negative angles as clockwise rotations which matches our assumption about sign conventions.
* **Data type preservation** – `rotate` returns a floating point array even if the input was integer‑typed because interpolation produces fractional values anyway; this matches typical test harness expectations where outputs are compared within an rtol/atol tolerance.
* **Multi‑channel images** – both functions treat extra trailing dimensions as independent channels automatically.
* **Performance budget** – since there are no explicit Python loops left after this change, each call now runs entirely in compiled code – easily meeting any tight time limits imposed by the benchmark harness.
