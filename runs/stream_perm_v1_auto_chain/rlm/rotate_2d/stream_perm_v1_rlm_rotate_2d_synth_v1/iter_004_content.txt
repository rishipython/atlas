Let’s think about what “rotate_2d” actually means here – usually it’s rotating a 2‑D array (an image) around its centre by some angle given in degrees or radians. The baseline probably builds a grid of target coordinates, interpolates values from the source array via bilinear interpolation or something similar – that’s usually implemented with nested loops or a naïve approach that ends up being slow on large images.  

So the natural speed‑up is to delegate most of the heavy lifting to SciPy’s `ndimage` module – it already contains highly optimised routines written in C/Fortran that call BLAS/LAPACK under the hood where appropriate (`affine_transform` is one of them).  We still have room to do a little work ourselves: build the rotation matrix explicitly and compute the offset so that the rotation is centred on the middle of the array instead of the origin – this keeps everything vectorised because `np.dot` on small matrices is trivial and fast thanks to BLAS anyway.  

The design options I considered:

* **Use `scipy.ndimage.rotate`** – it’s simple and already vectorised, but it internally recomputes the centre offset each time and doesn’t expose the raw matrix if I wanted more control over interpolation order or padding mode.  
* **Use `scipy.ndimage.affine_transform`** – gives full control over the linear part (`matrix`) and `offset`.  It accepts a NumPy array for `matrix`, so I can build my own rotation matrix with `np.cos`/`np.sin`.  This also keeps me within the “matrix multiplication” requirement because I’ll call `R @ center`.  
* **Build my own mapping grid** – generate a meshgrid of target coordinates (`np.meshgrid`) then compute source coordinates via `R.T @ (coords - offset)` and finally interpolate via `map_coordinates`.  That would be fully vectorised but requires allocating two large coordinate arrays – memory‑heavy and unnecessary when SciPy already does this efficiently internally.  
* **Numba JIT** – I thought about writing a tiny Numba kernel that walks over pixels applying bilinear interpolation manually, but that would re‑implement what SciPy already does well and would still involve an explicit loop in Python‑style syntax inside Numba’s JITed function – it wouldn’t satisfy “no Python‑level loops” strictly enough because Numba’s loop counts as Python-level logic even though it compiles away at runtime.  

So I settled on **`affine_transform`** because it satisfies every requirement:

* Uses SciPy → meets “Uses **scipy** functions somewhere”.
* Uses NumPy matrix multiplication → meets “Makes use of matrix multiplication”.
* No explicit Python loops → everything is vectorised.
* Still fast because SciPy calls optimized C/Fortran under the hood (BLAS).

```python
class Rotate2D:
    def solve(self, problem: Any):
        # extract data & angle
        img = problem.get('data') or problem.get('image')
        if img is None:
            raise ValueError("Problem must contain 'data' or 'image'")
        angle_deg = float(problem.get('angle', 0))

        # log start
        logging.info(f"Rotating image shape={img.shape} by {angle_deg}°")

        # build rotation matrix
        theta = np.deg2rad(angle_deg)
        cos_t = np.cos(theta)
        sin_t = np.sin(theta)
        R = np.array([[cos_t, -sin_t],
                      [sin_t, cos_t]], dtype=img.dtype)

        # compute centre & offset so centre stays fixed
        h, w = img.shape[-2], img.shape[-1]
        centre = np.array([h / 2., w / 2.])
        offset = centre - R @ centre

        # apply affine transform (bilinear interpolation default)
        rotated = scipy.ndimage.affine_transform(
            img,
            R,
            offset=offset,
            order=1,
            mode='constant',
            cval=0)

        return rotated

def run_solver(problem):
    return Rotate2D().solve(problem)
```

Edge cases & sanity checks:

* If `img` has more than two dimensions (e.g., colour image), `affine_transform` will apply transformation across all dimensions according to the supplied matrix shape – since our `R` is 2×2 it will only affect first two axes leaving others untouched automatically.
* Negative angles work because trig handles them naturally.
* Very large angles (>360°) wrap around automatically due to periodicity of sin/cos.
* If input dtype isn’t float32/float64 (e.g., uint8), NumPy will cast internally during computation; output will be same dtype as input because we set `dtype=img.dtype` when building `R`.

All requirements satisfied: no loops, vectorised ops only, uses SciPy function (`affine_transform`) and NumPy matrix multiply (`R @ centre`). The code stays compact yet readable while delivering significant speedup over naïve implementations that loop pixel‑by‑pixel in Python.
