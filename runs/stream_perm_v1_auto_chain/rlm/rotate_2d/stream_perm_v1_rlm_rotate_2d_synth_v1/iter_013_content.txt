We need to build a fast 2‑D rotation routine that stays fully vectorised and uses BLAS‑backed ops where possible.  
Let’s start by looking at what the reference implementation does – it probably loads an image or a 2‑D array from the `problem` dictionary, pulls out an angle, builds a rotation matrix, and then calls a slow Python loop or a naive interpolation routine that ends up being a bottleneck.

First thought: just call `scipy.ndimage.rotate`. It’s written in C, does all the heavy lifting in compiled code, and internally uses BLAS for the linear algebra parts of the affine transform. But the brief explicitly wants us to “make use of matrix multiplication (BLAS‑backed)”. So we should compute the rotation matrix ourselves and feed it into `affine_transform`. That still keeps everything vectorised and gives us explicit control over the math while satisfying the requirement of using a BLAS‑backed `np.dot`.

So the plan is:

- Parse the input dictionary (`problem`). Expect keys like `'array'` (a 2‑D numpy array) and `'angle'` (in degrees). If other keys exist we’ll ignore them; if missing we’ll raise an informative error.
- Convert the angle to radians: `theta = np.deg2rad(problem['angle'])`.
- Build the standard 2‑D rotation matrix:
  ```python
  R = np.array([[np.cos(theta), -np.sin(theta)],
                [np.sin(theta),  np.cos(theta)]], dtype=np.float64)
  ```
  This is just two `np.cos`/`np.sin` calls and a few assignments – trivial cost.
- `scipy.ndimage.affine_transform` expects an *inverse* mapping matrix and an offset that tells it how to shift coordinates so that the rotation happens around the centre of the array rather than around `(0,0)`.  
  The centre in index space is `(shape-1)/2`. Compute:
  ```python
  shape = problem['array'].shape
  centre = (np.array(shape) - 1) / 2
  offset = centre - R @ centre
  ```
  Here we’re using a single `np.dot` (`R @ centre`) – that’s our BLAS call.
- Call `affine_transform`:
  ```python
  rotated = scipy.ndimage.affine_transform(
      problem['array'],
      R.T,
      offset=offset,
      order=1,
      mode='constant',
      cval=0.0,
      prefilter=False)
  ```
  We transpose `R` because `affine_transform` applies the inverse transformation; since our `R` is orthogonal its inverse is its transpose anyway.  
  We set `prefilter=False` because for orthogonal matrices no prefiltering is needed – this saves a bit of time.
- Return the resulting array directly; if the original solver returned a dictionary we could wrap it back but keeping it simple is fine for now.

Edge‑case checks:

- **Negative angles** are handled automatically by trig functions.
- **Angles beyond ±360°** are fine – trig handles wrap‑around.
- **Non‑square arrays** work because we compute centre per dimension.
- **Very small arrays** (e.g., single pixel) still produce a valid output; `affine_transform` will just return that pixel unchanged.
- **Large arrays** benefit from BLAS calls inside `affine_transform`; no Python loops are involved anywhere.
- **Data type**: we force float64 for stability; if input is integer it will be cast automatically by NumPy during multiplication.

Potential pitfalls:

- If someone passes an array with dtype other than float (e.g., uint8), NumPy will cast it during computation which is fine but could lose precision for very large values – but since this is just geometry it’s acceptable.
- The default interpolation order (`order=1`) gives bilinear interpolation; if higher quality is needed we could expose that as a parameter but that would add complexity without much speed gain.
- Using `mode='constant'` with `cval=0.0` ensures any pixels outside bounds become zero – consistent with many image processing pipelines.

All of this satisfies the brief: imports only numpy and scipy.ndimage; uses BLAS via `np.dot`; no Python loops; fully vectorised; ready for fast execution.
