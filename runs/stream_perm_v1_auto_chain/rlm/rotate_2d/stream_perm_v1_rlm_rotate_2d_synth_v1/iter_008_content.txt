### Design decisions

#### 1. Detecting input type
We’ll inspect `problem`.  
If `'points'` key exists → treat as point set; otherwise if `'image'` key exists → treat as image.  
If neither present → raise a ValueError – this keeps us honest about expectations.

#### 2. Point rotation
```python
angle_rad = np.deg2rad(problem['angle'])   # assume degrees in spec
c, s = np.cos(angle_rad), np.sin(angle_rad)
R = np.array([[c, -s], [s, c]], dtype=problem['points'].dtype)
rotated = problem['points'] @ R.T
```
No loops at all – just one BLAS‑backed matmul (`@`).  
The result keeps the same dtype as input which avoids unnecessary casts.

#### 3. Image rotation
```python
rotated_img = scipy.ndimage.rotate(
    problem['image'],
    angle=problem['angle'],
    reshape=False,
    order=1   # bilinear interpolation – fast enough for most tasks
)
```
This single call hits compiled C code under SciPy/Numpy; no Python overhead beyond passing arguments.

#### 4. Logging
A few concise log statements at entry/exit of `solve`.  
They’re cheap because they’re executed only once per call:

```python
logging.debug("Rotate2D.solve called with %s", type(problem).__name__)
...

```

#### 5. Return format
Return a dictionary mirroring what the baseline expects – e.g.,

```python

{'rotated_image': rotated_img}       # for image case
```

If both keys are present we return both results; this makes the function more flexible without breaking callers that expect only one output type.

#### 6. Edge cases & robustness
* **Zero angle** → identity matrix → dot product yields original array instantly.
* **Negative angles** work automatically via trigonometric identities.
* **Large arrays** are handled by NumPy/SciPy’s memory‑efficient routines; no explicit copying beyond what those libraries do internally.
* **Data types** – we preserve the original dtype for points; images are passed straight through SciPy which preserves dtype unless interpolation changes it (bilinear stays float).
