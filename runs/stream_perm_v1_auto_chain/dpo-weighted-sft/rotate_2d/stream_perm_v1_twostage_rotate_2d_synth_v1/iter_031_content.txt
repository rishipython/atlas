def run_solver(problem):

But before we lock this in let’s consider a few alternative designs that might look tempting:

1. **Use `cv2.warpAffine` from OpenCV**  
   OpenCV can be very fast because it calls into highly optimised libraries on GPU/CPU depending on build flags. However OpenCV isn’t listed among the allowed packages – only `scipy`, `numpy`, etc., are guaranteed available in this environment – so this path is off the table unless we ship OpenCV ourselves which would violate the “single file” rule and likely break import resolution on the judge platform.
   
2. **Use `scipy.signal.convolve` + FFT**  
   One could attempt a clever trick: compute the Fourier transform of the image and multiply by a phase ramp corresponding to rotation in Fourier space (a shear‑based approach). That would avoid interpolation entirely but it would require implementing several non‑trivial steps (FFT shift, shearing matrices) and would still involve Python loops for handling multiple shears unless we hand‑craft everything in one big kernel – which becomes messy and hard to debug.
   
3. **Write a Numba JIT kernel**  
   A straight‑forward Numba implementation could give decent speedups over pure Python loops because it compiles tight loops into machine code at runtime. However Numba still incurs overhead for each call (especially for small arrays) and it won’t help if the bottleneck is actually memory bandwidth rather than CPU compute – which is usually the case for image rotation.
   
4. **Use `skimage.transform.rotate`**  
   This library offers more flexibility (different interpolation orders) but under the hood it still calls `ndimage.rotate`. It also adds a dependency on scikit‑learn/scikit‑image which isn’t part of the allowed packages list – so we’ll skip it.
   
5. **Hand‑rolled bilinear interpolation**  
   We could write a fully vectorised bilinear interpolation routine using only NumPy broadcasting (`np.floor`, `np.modf`, indexing). While theoretically possible without explicit loops (by constructing coordinate grids), this gets messy quickly when you have to handle boundary conditions correctly (clamp vs wrap vs reflect). It would also likely be slower than calling into C libraries because NumPy’s broadcasting still incurs Python overhead per element operation.
   
After weighing these options, `scipy.ndimage.rotate` emerges as the cleanest path: it satisfies all constraints (uses SciPy, no Python loops), is battle‑tested across many backends (OpenBLAS / MKL), supports different interpolation orders (`order=0…5`), handles reshaping automatically when you want an output large enough to contain the whole rotated image (`reshape=True`), and is already heavily optimised in C/C++ so you get maximum speed without writing any low‑level code yourself.

Now let’s think about edge cases that could trip us up:

- **Zero or full‑rotation angles** – rotating by exactly 0°, 360°, or multiples thereof should just return a copy of the original array without invoking any heavy machinery.
- **Negative angles** – `ndimage.rotate` accepts negative values naturally so no special handling required.
- **Non‑float input types** – we’ll cast everything to `float64` via `np.asarray`; this ensures consistent behaviour across integer inputs while keeping precision high.
- **Empty arrays** – an empty input should simply propagate an empty output; calling `rotate` on an empty array raises an error in some SciPy versions, so we guard against that explicitly.
- **Large images** – memory consumption scales linearly with image size; there’s no way around that except possibly chunking which would reintroduce loops – so we’ll leave it as is.
- **Interpolation order** – defaulting to cubic (`order=3`) gives good visual quality for most applications while still being reasonably fast; users who need nearest neighbour can override via an optional key in `problem`.

import logging
from typing import Any, Dict
import numpy as np
import scipy.ndimage

# Configure a logger for debugging if needed
logger = logging.getLogger(__name__)
logger.setLevel(logging.INFO)

def run_solver(problem: Dict[str, Any]) -> np.ndarray:
    """
    Rotate a 2-D array according to ``problem`` specifications.
    
    Parameters
    ----------
    problem : dict
        Must contain at least:
            - ``matrix`` : 2-D sequence convertible to ndarray
            - ``angle`` : rotation angle in degrees (float)
        Optional keys:
            - ``order`` : interpolation order (int 0–5)
            - ``reshape`` : whether output should be resized (bool)
    
    Returns
    -------
    ndarray
        Rotated version of ``matrix``.
    
    Notes
    -----
    The function is fully vectorised – all heavy lifting happens inside SciPy’s C code.
    
    """
    
    # Extract mandatory fields safely
    try:
        arr = np.asarray(problem["matrix"])
        ang = float(problem["angle"])
    except KeyError as exc:
        raise ValueError(f"Missing required key {exc}") from None
    
    # Optional parameters with sane defaults
    order = int(problem.get("order", 3))
    reshape = bool(problem.get("reshape", True))
    
    # Quick path for trivial rotations
    if ang % 360 == 0 or arr.size == 0:
        logger.debug("Angle %s yields identity transformation.", ang)
        return arr.copy()
    
    # Perform rotation using SciPy's highly optimised routine
    logger.debug(
        "Rotating array of shape %s by %s degrees "
        "(order=%s, reshape=%s).",
        arr.shape,
        ang,
        order,
        reshape,
    )
    
    rotated = scipy.ndimage.rotate(
        arr,
        ang,
        reshape=reshape,
        order=order,
        mode="constant",
        cval=0,
        prefilter=True,
    )
    
    logger.debug("Rotation complete.")
    
    return rotated
