### Understanding the input

From previous experience with AlgoTune problems, the `problem` argument is usually a dictionary containing at least two keys:

```python
{
    'matrix': <numpy array>,   # or 'data'
    'angle': <float>,          # rotation angle in degrees
}
```

The baseline implementation probably pulls these values out in some way and then calls `Rotate2D().solve`. We’ll mirror that pattern:

```python
def run_solver(problem):
    # Extract matrix
    mat = np.asarray(problem.get('matrix') or problem.get('data'))
    # Extract angle
    theta = float(problem.get('angle', 0))
```

If neither key exists we raise a clear error; this keeps the behaviour deterministic for the benchmark harness.

---

### Rotating with `scipy.ndimage`

`scipy.ndimage.rotate` accepts:

- The input array.
- The rotation angle in degrees.
- Optional parameters:
  - `reshape`: whether to enlarge the output so that the whole rotated image fits. The baseline likely keeps the same shape (common for image processing), so we set `reshape=False`.
  - `order`: interpolation order (0=nearest, 1=bilinear, 3=cubic). The reference might use nearest‑neighbor; we’ll choose `order=1` for a good trade‑off between speed and quality while still being fast because it’s implemented in C.
  - `mode`: how to handle boundaries; default `'constant'` with cval=0 works fine for most benchmarks.

So:

```python
rotated = scipy.ndimage.rotate(mat, theta, reshape=False, order=1)
```

No loops here – all work happens inside the compiled routine.

### Returning the result

The baseline likely returns just the rotated array (or maybe wrapped in a dict). To stay compatible we’ll return it directly:

```python
return rotated
```

If the benchmark expects a dict with a specific key (`'result'`) we can wrap it accordingly; but since we don’t see that detail we’ll just return the array itself – this matches typical usage in these tasks.

### Edge‑case checks

- **Zero or negative angles** – handled naturally by `rotate`.
- **Non‑2D arrays** – if shape has fewer than two dimensions raise ValueError; if more than two dimensions treat only first two axes (though unlikely).
- **Large arrays** – memory usage stays within limits because we never create extra copies beyond what `rotate` does internally.
- **Different dtypes** – output dtype matches input dtype automatically unless interpolation changes it; using order=1 keeps it float64 for integer inputs unless cast back explicitly if needed (baseline probably keeps original dtype).
