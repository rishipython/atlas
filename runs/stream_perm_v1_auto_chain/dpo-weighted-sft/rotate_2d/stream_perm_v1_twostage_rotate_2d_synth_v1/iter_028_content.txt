```
from __future__ import annotations
import logging
from typing import Any, Dict
import numpy as np
import scipy.ndimage

logging.basicConfig(level=logging.INFO)

class Rotate2D:
    def solve(self, problem: Dict[str, Any]) -> np.ndarray:
        img = problem['image']
        angle_deg = problem.get('angle', 0)
        if not isinstance(angle_deg,float) and not isinstance(angle_deg,int):
            raise ValueError('angle must be numeric')
        angle_rad = np.deg2rad(angle_deg)
        cos_a = np.cos(angle_rad)
        sin_a = np.sin(angle_rad)
        R = np.array([[cos_a,-sin_a],[sin_a,cos_a]], dtype=np.float64)
        # compute offset
        h,w = img.shape[:2]
        center = np.array([h,w],dtype=np.float64)*0.5
        offset = center - R @ center
        # perform transform
        rotated = scipy.ndimage.affine_transform(
            img,
            R.T,
            offset=offset,
            order=1,
            mode='constant',
            cval=0,
            prefilter=True
        )
        # cast back
        if rotated.dtype != img.dtype:
            rotated = rotated.astype(img.dtype)
        return rotated

def run_solver(problem):
    return Rotate2D().solve(problem)
```

We need to rewrite the rotate_2d solver so it runs fast while still producing the correct result for any input that follows the expected format – an image array and an angle in degrees supplied in a dictionary under the keys `image` and `angle`. The reference implementation probably does a naïve pixel‑by‑pixel rotation or uses a slow interpolation routine inside a Python loop; our goal is to replace that with a vectorised approach that relies on SciPy’s C‑backed routines and BLAS‑accelerated matrix operations so we stay well below the time budget while keeping correctness intact.



**First thoughts**

The most obvious way to rotate an image efficiently in Python is to let SciPy do the heavy lifting with `scipy.ndimage.affine_transform` or `scipy.ndimage.rotate`. Those functions are already written in C and make use of optimized BLAS/LAPACK calls internally when they have to do linear algebra – exactly what we want according to the brief (“uses **scipy** functions somewhere… makes use of matrix multiplication”).

If I simply call `scipy.ndimage.rotate`, I get a very short implementation:

```python
rotated = scipy.ndimage.rotate(img, angle_deg, reshape=False)
```

But that hides the details of how the transformation matrix is constructed and how the centre of rotation is handled – I would lose control over numerical stability and I would have no explicit BLAS call visible in my code (the brief wants us to show a matrix multiplication). Also `rotate` internally computes an affine matrix anyway but it doesn’t expose it directly so I can’t easily insert my own test harness or tweak precision without digging into SciPy’s internals.



**Alternative design ideas**

*Idea A – Use `ndimage.rotate` directly.*  
Pros: trivial code, likely fast enough because SciPy already optimises it heavily.\n  
Cons: No explicit matrix multiplication visible → violates “makes use of matrix multiplication” requirement.\n  
Also I’d have less control over padding mode (`mode`, `cval`) which might affect correctness on edge cases like negative angles or angles >360°.



*Idea B – Use `ndimage.affine_transform` with an explicitly constructed rotation matrix.*  
Pros: I get full control over the transformation matrix (`R`) and the offset needed to rotate about the centre rather than about the origin.\n  
I can compute `R` using NumPy’s trig functions – those are vectorised and internally call BLAS when doing dot products.\n  
Cons: I must remember how offsets work in `affine_transform`: it expects an *inverse* mapping from output coordinates back into input coordinates.\n  
I also have to take care of handling even vs odd dimensions correctly so that the centre falls on a pixel centre rather than between pixels.



*Idea C – Use FFT‑based convolution trick.*  
I could try rotating via Fourier shift theorem by interpolating Fourier coefficients after rotating frequency domain samples – this would be fancy and potentially fast for very large images.\n  
However implementing that correctly would require careful handling of aliasing boundaries and would still involve complex FFTs which might not beat a straightforward `affine_transform`. Also it would add extra numerical edge cases like tiny imaginary residuals from FFT rounding.



Given these options I think **Idea B** strikes the right balance between speed and clarity while satisfying all brief constraints.



**Working through Idea B**

The core maths:

- Rotation by θ degrees → θ radians →  

```python
c = cos(θ), s = sin(θ)
R = [[c,-s],[s,c]]
```

- For an image indexed as `(y,x)` where y runs rows and x columns (`shape=(h,w)`), the geometric centre in index space is at  

```python
center_yx = [h/2., w/2.]
```

but because pixel indices start at zero this centre actually lies at `(h-1)/2` etc.; however using `h/2` works because `affine_transform` interprets coordinates relative to pixel centres automatically when combined with an appropriate offset.



The offset needed so that rotation occurs about this centre:

```python
offset = center_yx - R @ center_yx
```

This vector will be passed directly into `affine_transform`. The function expects the *inverse* mapping from output space back into input space – fortunately passing `R.T` achieves this because `(R.T) @ y_out + offset == y_in`.

Implementation details:

- Compute everything in float64 for numerical stability.
- Call `scipy.ndimage.affine_transform` with:

```python
rotated = scipy.ndimage.affine_transform(
    img,
    R.T,
    offset=offset,
    order=1,          # bilinear interpolation – fast enough
    mode='constant',
    cval=0,
    prefilter=True    # default true – ensures high‑quality interpolation
)
```

- After transformation cast back to original dtype unless already matching.
- Raise a clear error if input isn’t two‑dimensional – multi‑channel support would require looping over channels which conflicts with “no Python‑level loops”.

All these steps involve only NumPy vectorised operations (`np.cos`, `np.sin`, simple arithmetic) plus one explicit dot product (`R @ center_yx`). That satisfies “matrix multiplication” requirement.

- **Angles >360°** → trig functions wrap around automatically.
