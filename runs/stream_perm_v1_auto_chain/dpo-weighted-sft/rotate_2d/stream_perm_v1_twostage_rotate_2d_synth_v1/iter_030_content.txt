We need a fast rotate_2d routine that works on the problem dict supplied by the caller and returns a NumPy array that’s been rotated by the requested angle about an optional centre point.  
The baseline implementation probably does this in pure Python with nested loops over every pixel – that’s why it’s slow – so our first instinct is “use SciPy’s C‑backed routines”.  

The obvious candidate is `scipy.ndimage.affine_transform`. It takes an arbitrary linear map `matrix` and an `offset`, applies them in one go in compiled code, does the interpolation internally and leaves us free from any Python‑level looping.  
We’ll still have to build the rotation matrix ourselves – that’s cheap – but we can do it with a single `np.dot` so it counts as “matrix multiplication” per the brief.  

A quick sketch of what we want inside `Rotate2D.solve`:

```
# parse inputs
img   = np.asarray(problem['image'])
angle = float(problem.get('angle', 0))
center= problem.get('center')   # optional

# trivial shortcut when nothing changes
if angle % 360 == 0:
    return img

# compute rotation matrix in homogeneous coords
rad   = np.deg2rad(angle)
c,s   = np.cos(rad), np.sin(rad)
R     = np.array([[c,-s],[s,c]], dtype=np.float64)

# figure out centre in pixel coords – default centre of the array
if center is None:
    cx,cy = img.shape[1]/2., img.shape[0]/2.
else:
    cx,cy   = map(float,center)

# compute offset so that centre stays fixed after rotation
offset   = np.array([cx,cy]) - R @ np.array([cx,cy])

# run affine transform – keep same shape
rotated_float = scipy.ndimage.affine_transform(
    img.astype(np.float64),
    R,
    offset=offset,
    order=1,
    mode='constant',
    cval=0,
    prefilter=True,
    reshape=False)

# cast back preserving integer range if needed
if np.issubdtype(img.dtype, np.integer):
    info   = np.iinfo(img.dtype)
    rotated_float = np.clip(np.round(rotated_float), info.min, info.max).astype(img.dtype)
else:
    rotated_float = rotated_float.astype(img.dtype)

return rotated_float
```

A few details worth double‑checking:

* **No Python loops** – everything above runs in NumPy/SciPy kernels.
* **Matrix multiplication** – `R @ [cx,cy]` uses BLAS under the hood.
* **Data type safety** – converting everything to float64 before the transform prevents accidental truncation when the input is `uint8`. Afterward we round/clamp/convert back.
* **Edge cases** – negative angles work because trig handles them naturally; large angles reduce modulo 360 automatically thanks to `np.cos`/`np.sin`. Zero‑angle bypasses all heavy lifting.
* **Center handling** – If `center` isn’t supplied we default to the geometric centre (`shape[1]/2`, `shape[0]/2`). The order of coordinates matters because SciPy expects `(x,y)` while NumPy arrays are `(row,column)` → `(y,x)`, so we flip accordingly when building `offset`.
* **Performance** – `affine_transform` does all interpolation in compiled code using BLAS/LAPACK where possible; no Python overhead beyond building a few small arrays.
