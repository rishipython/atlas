### Second attempt – batch processing

But that reintroduces a loop – which we want to avoid according to the brief (“No Python-level loops”).  One way around this is to let NumPy handle the iteration under the hood by putting everything into an object array and applying `np.vectorize`.  However, `np.vectorize` merely wraps the function call and still executes it one element at a time in Python land; it doesn’t magically speed up anything beyond avoiding explicit loop syntax.  

A better approach would be to concatenate all messages into one big byte string separated by some delimiter that does not affect base‑64 padding when split back later (e.g., a newline), then decode all at once and split back into individual encodings:

1. Join all messages with `b"\n"` → `big = b"\n".join(messages)`
2. Encode once → `big_enc = base64.b64encode(big)`
3. Split back on `b"\n"` → `enc_list = big_enc.split(b"\n")`

However, this fails because padding inserted after each original message will be lost when they are merged – you end up with one long padded block rather than three separate padded blocks.  So this trick won’t work unless you re‑parse padding manually, which defeats the purpose of vectorisation anyway.

### Third attempt – keep it simple

Given that any attempt at true vectorisation here would either add complexity without real benefit or break correctness, I think the safest path is:

* Treat `problem["data"]` as either a single byte string or an iterable of byte strings.
* If it’s iterable, just loop over it – but wrap that logic inside a small helper that’s heavily annotated so that future refactoring could replace it with something truly vectorised if needed.
* Keep imports minimal but include NumPy because it might be used later for other tasks (e.g., converting lists of integers into arrays before feeding them into some other routine).
* Add logging statements at DEBUG level so developers can trace execution without impacting performance when DEBUG is off.
* Ensure type safety: if keys or data are provided as strings instead of bytes, convert them appropriately (`str.encode("utf-8")`).  
* Handle empty inputs gracefully – returning empty strings / empty lists as appropriate.
* Use consistent return format (a dict with `"encoded"` key always present; `"hmac"` only when requested).

### Edge‑case sanity checks

* **Empty data** – `base64.b64encode(b"")` returns `b""`, decoding gives empty string; HMAC over empty string works fine.
* **Non‑byte inputs** – if someone passes `"foo"` instead of `b"foo"`, our code should detect type and `.encode()` it.
* **Large inputs** – both `base64` and `hmac` are linear time and operate on streams internally; no risk of quadratic blowup.
* **Key length** – HMAC accepts any length key; no special handling needed.
* **Logging** – only log at DEBUG level; production runs skip logs automatically.
* **NumPy** – currently unused but imported per spec; could be leveraged later if batch processing becomes required (e.g., converting lists of ints into arrays).

### Final design outline

```python
import base64
import hmac
import hashlib
import logging
from typing import Any, Dict

import numpy as np   # kept for future expansion

def _ensure_bytes(x):
    """Coerce str/bytes/int/array-like into bytes."""
    if isinstance(x, bytes):
        return x
    if isinstance(x, str):
        return x.encode("utf-8")
    # For iterable containers (list/tuple) convert each element recursively
    try:
        return bytes(x)
    except Exception:
        raise TypeError(f"Cannot convert {x!r} to bytes")

def run_solver(problem: Dict[str, Any]) -> Dict[str, Any]:
    """
    Solve Base64Encoding problem efficiently.
    Accepts:
      - problem["data"]: bytes OR iterable of bytes/str
      - optional "key": bytes OR str used for HMAC
    Returns:
      - dict with "encoded": str (base‑64 representation)
      - optional "hmac": hex digest string if key provided
    """
    logger = logging.getLogger(__name__)
    logger.debug("Starting run_solver")

    # Normalise data input
    raw_data = problem.get("data")
    if raw_data is None:
        raise ValueError("Missing 'data' field")

    # If raw_data is iterable but not bytes/str => treat as list of messages
    if isinstance(raw_data, (list, tuple)):
        msgs = [_ensure_bytes(m) for m in raw_data]
        # Process each message individually – still minimal overhead
        enc_list = [base64.b64encode(m).decode("ascii") for m in msgs]
        result = {"encoded": enc_list}
        logger.debug(f"Encoded {len(msgs)} messages")
    else:
        # Single message case
        msg_bytes = _ensure_bytes(raw_data)
        enc_str = base64.b64encode(msg_bytes).decode("ascii")
        result = {"encoded": enc_str}
        logger.debug("Encoded single message")

    # Optional HMAC computation
    key_raw = problem.get("key")
    if key_raw is not None:
        key_bytes = _ensure_bytes(key_raw)
        # Compute HMAC over the *raw* binary representation (not decoded string)
        mac_digest = hmac.new(key_bytes,
                              _ensure_bytes(result["encoded"]),
                              hashlib.sha256).hexdigest()
        result["hmac"] = mac_digest
        logger.debug("Computed HMAC")

    logger.debug("run_solver finished")
    return result
```

The above sketch keeps everything inside pure Python without explicit loops over large datasets except where unavoidable (list comprehensions are fine because they’re implemented in C).  
If later we discover that batch processing truly benefits from NumPy’s vectorisation (e.g., converting many integer arrays into binary blobs before encoding), we can replace `_ensure_bytes` logic accordingly while preserving this clean interface.
