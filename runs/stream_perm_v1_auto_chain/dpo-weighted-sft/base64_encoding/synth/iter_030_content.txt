We need a fast Base‑64 encoder that works on the whole input in one shot – no Python loops at all – and still follows the same API as the reference implementation (`Base64Encoding.solve(problem)` returning a string).  
The input comes in a `problem` dictionary under the key `"data"`. It could be a `bytes`/`bytearray`, a UTF‑8 string or even a list/tuple/NumPy array of integers in the range `0–255`.  
First thing is normalising that payload into a flat `np.ndarray` of type `uint8`.  
```python
data_inp   : any type from the dictionary
data_bytes : normalised `bytes`
```
If it’s already `bytes`/`bytearray` just keep it – otherwise encode strings or build a `bytes` object from lists/tuples/arrays.  
Once we have `data_bytes`, the rest is pure NumPy gymnastics:

* Compute how many padding bytes are required (`pad_len`).  
* Pad with zeros up front so every chunk has exactly three bytes – this keeps the logic simple and lets us use vectorised bit‑shifts without branches.
* Reshape into an `(N//3 , 3)` view (`uint32` so shifts don’t overflow).
* Build one big `uint32` value per chunk (`int24`) by combining the three octets.
* Extract four six‑bit groups per chunk with right shifts and masks (`& 0x3F`).  
* Stack those four indices together (`shape (-1,)`) – now each element is an integer in `[0..63]`.
* Map those integers to ASCII codes using a pre‑computed lookup table (`alphabet_codes`).  
* Convert the resulting numeric array back into raw bytes (`tobytes()`).
* Finally overwrite the last `pad_len` characters with `'='` *before* decoding back into a Python string – this keeps everything vectorised and avoids any Python‑level slicing logic after the fact.
```python
alphabet_codes : np.ndarray(uint8) holding ASCII codes for “ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/”
```

The only non‑vectorised part is that tiny slice assignment when applying padding – it’s just one write on an already allocated NumPy buffer and costs nothing compared with the bulk operations above.  

All other steps are single BLAS‑style calls under the hood (`reshape`, bitwise ops, indexing), so they run entirely in C/BLAS land without any Python interpreter involvement after the initial setup of the arrays – exactly what we want for speedup over the pure‑Python reference implementation that loops over every byte or every group of three manually.



**Edge cases**  
* Empty input → return `''` immediately – all downstream NumPy shapes handle zero length gracefully anyway.
* Input length already divisible by three → `pad_len == 0`; our logic skips any padding replacement.
* Very large inputs → memory footprint is roughly twice the size of the input plus a few small buffers – well within typical limits for competitive programming problems.
