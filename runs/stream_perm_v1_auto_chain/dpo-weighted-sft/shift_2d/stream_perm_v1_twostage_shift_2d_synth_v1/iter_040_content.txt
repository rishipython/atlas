We need to replace the slow Shift2D solver with a fast vectorised implementation that still honours the interface `run_solver(problem)` and returns an optimal solution quickly while staying within the constraints given in the brief – no Python loops, heavy reliance on NumPy/Scipy functions, a handful of imports including `logging`, `random`, `typing`, `numpy`, `scipy.ndimage`.

First thought was a straight‑forward wrapper around `scipy.ndimage.shift`. That gives us a single BLAS‑level call per image – perfect speedwise – so I’ll start there. The only wrinkle is that the baseline might accept images with more than two dimensions (e.g., colour images with shape `(H,W,C)` or even `(B,H,W,C)` for batches). The `ndimage.shift` function expects a `shift` tuple whose length matches the number of dimensions of the array being shifted. If the caller supplies just a two‑element tuple `(dy,dx)` for a three‑dimensional array we’ll get an error because it will try to apply that pair of shifts along all three axes instead of leaving the third axis untouched. To avoid this I’ll normalise the supplied `shift` argument:

* If the user passes a scalar → broadcast it across all axes.
* If they pass a sequence whose length matches `arr.ndim` → use it verbatim.
* If they pass a sequence whose length is one less than `arr.ndim` → append a zero so that the extra axis stays fixed.
* Anything else → raise an informative error so callers know what they did wrong.

With that logic in place I can safely call:

```python
scipy.ndimage.shift(arr,
                    shift=full_shift,
                    mode=mode,
                    order=order)
```

The parameters `mode` and `order` are optional in the problem dict – defaulting respectively to `'nearest'` and cubic interpolation (`order=3`). Using cubic gives good visual quality while still being fast enough for most image sizes; users who care about speed can override it if they wish.

I also want to satisfy the “no Python‑level loops” rule strictly – everything from reading the input dictionary through converting it into a NumPy array down to calling `ndimage.shift` happens in compiled code or via NumPy’s own vectorised routines. No explicit loops are written in Python anywhere in this function body except possibly very small sanity checks which are trivial overhead compared to the heavy lifting done inside SciPy’s C implementation.
