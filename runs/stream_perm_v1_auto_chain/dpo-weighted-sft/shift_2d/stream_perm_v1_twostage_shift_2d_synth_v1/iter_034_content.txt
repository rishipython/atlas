Let’s first sketch what “shift_2d” actually means in this context. The reference implementation probably takes a dictionary containing a 2‑D NumPy array (the image or data field) under a key such as `"data"` and a pair of floating‑point offsets under `"shift"`. The goal is simply to translate the array by that offset – potentially fractional – using bilinear interpolation so that sub‑pixel accuracy is preserved while keeping the output shape unchanged (padding with zeros where necessary).  

The naive baseline most likely iterates over every pixel in Python, computes its source coordinate via inverse mapping `(x−dx,y−dy)`, samples from the input array (maybe with linear interpolation) and writes into an output array – all done in pure Python nested loops or list comprehensions – which is painfully slow on anything larger than a few hundred pixels per dimension.  

To speed this up we should delegate the heavy lifting entirely to compiled C/Fortran code in SciPy/Numpy rather than reimplementing it ourselves in Python or NumPy alone (NumPy’s broadcasting won’t help here because we need sub‑pixel sampling).  

### Candidate libraries / functions

* **`scipy.ndimage.shift`** – built exactly for this job; it accepts an array, a tuple of offsets `(dy,dx)` (note the order), an `order` parameter controlling spline interpolation (`order=1` gives bilinear), a `mode` controlling border handling (`mode='constant', cval=0`) which pads zeros outside the original bounds – exactly what we want for a “shift” operation without cropping.
* **`scipy.ndimage.map_coordinates`** – more general mapping tool that could implement any arbitrary coordinate transform via spline interpolation (`order=1`). It requires us to build coordinate grids explicitly which introduces extra memory traffic compared with `shift`.
* **Numba JIT** – could write our own looped version that gets compiled away, but this still involves writing a loop body even though it runs in machine code – against the brief’s “no Python‑level loops” rule (the rule applies at source‑code level).
* **FFT convolution** – would be overkill here because shifting is a simple translation operation that doesn’t benefit from frequency domain tricks unless you’re doing many different shifts on the same data repeatedly – not our case here.
* **Custom Cython** – again unnecessary complexity given `ndimage.shift` already does what we need efficiently in C/C++ under the hood.
  
So the cleanest choice is `scipy.ndimage.shift`. It’s already vectorised internally and does exactly what we want without any explicit looping in our own code path.  

### Handling edge cases & numerical niceties

* **Zero shift** – calling `shift` when both offsets are zero still incurs overhead of entering the C routine and allocating a new array copy even though nothing changes. We can guard against this trivial case with a quick `if dy == 0 and dx == 0:` branch that simply returns `img.copy()` (or even `img` itself if immutability isn’t an issue). This branch contains no loop – just a couple of comparisons.
* **Data type** – `ndimage.shift` promotes integer arrays up through float64 internally when doing spline interpolation so there’s no risk of overflow or truncation errors on integer inputs.
* **Negative / large offsets** – handled automatically by padding zeros (`cval=0`) when sampling outside bounds.
* **Clipping tiny negatives** – linear interpolation between non‑negative pixels should never produce negative values unless there’s floating‑point round‑off below machine epsilon (~1e‑16). If tests are extremely strict about exact zeros at padded borders one could do `np.maximum(result, 0)` afterwards – this is still vectorised because NumPy applies the comparison over the whole array in C without any Python loop.
