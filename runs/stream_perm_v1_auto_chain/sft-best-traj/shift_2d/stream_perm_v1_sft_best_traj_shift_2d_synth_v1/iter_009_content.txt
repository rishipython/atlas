But before locking into that I should consider alternatives:

1. **Pure NumPy roll + linear interpolation**  
   We could do an integer roll using `np.roll` and then apply a fractional offset via convolution with a small kernel (e.g., bilinear weights). That would avoid SciPy entirely but would require manual implementation of interpolation which is error‑prone and likely slower than SciPy’s optimized routines anyway because each pixel would still have Python‑level logic unless fully vectorised across the whole image – which is possible but more complicated than simply calling `ndimage.shift`. Also handling boundary conditions would be tricky – we'd have to decide on padding strategy manually (`mode='constant'` vs `'nearest'`).  

2. **FFT‑based phase multiplication**  
   A classic trick for sub‑pixel shifting is multiplying the Fourier transform by a linear phase ramp:  
   ```python
   F = np.fft.rfft2(img)
   u = np.arange(img.shape[1]) / img.shape[1]
   v = np.arange(img.shape[0]) / img.shape[0]
   phase = np.exp(-2j*np.pi*(u*dx + v[:,None]*dy))
   img_shifted = np.real(np.fft.irfft2(F * phase))
   ```  
   This approach is O(N log N) and can be very fast on large images if you already have FFT plans cached (e.g., via PyFFTW). However there are several pitfalls:
   - It assumes periodic boundary conditions – any non‑periodic content will wrap around incorrectly unless you pad first.
   - Padding introduces extra memory overhead and additional FFTs if you want true zero padding outside the original support.
   - For small images or modest shifts this overhead might outweigh any speed benefit.
   - Also implementing this correctly requires careful handling of frequency grids (`np.meshgrid`) and ensuring shapes match up.
   
3. **Numba JIT compiled loops**  
   One could write a JIT‑compiled function that iterates over pixels in C‑style loops inside Numba’s `@njit`. This eliminates Python overhead entirely and can achieve near‑C performance while still allowing arbitrary interpolation logic inside the loop (e.g., bilinear). The downside:
   - Still effectively a loop at C level – though JIT compiled it’s fast.
   - Requires careful management of bounds checks (`nopython=True`).
   - Slightly more complex code than calling an existing SciPy routine.
   
4. **Using JAX**  
   With JAX you could write a pure functional implementation that gets JIT compiled on GPU/CPU automatically via XLA. However setting up JAX kernels for arbitrary shapes can be overkill here; plus JAX doesn’t provide an out‑of‑the‑box subpixel shifting routine analogous to `ndimage.shift`. You’d end up writing your own interpolation logic anyway.
   
5. **scipy.signal.wiener or convolve**  
   Not relevant – those are filtering functions rather than shifting.
   
After weighing these options I think sticking with `scipy.ndimage.shift` is both simplest and fastest given our constraints:

- Boundary handling is straightforward (`mode='constant'`, `cval=0` gives zero padding).
- The API is simple: one function call with `(shift_y, shift_x)` tuple.
