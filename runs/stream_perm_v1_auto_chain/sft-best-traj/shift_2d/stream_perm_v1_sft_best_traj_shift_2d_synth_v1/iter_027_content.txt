        # extract arrays
        A = np.asarray(problem['A'])
        B = np.asarray(problem['B'])
        # ensure same shape
        H,W = A.shape
        if B.shape != (H,W):
            # pad B or crop

        # compute cross-correlation via FFT
        # Use np.fft.fftn for 2D
        fA = np.fft.rfftn(A)
        fB = np.fft.rfftn(B)
        corr = np.fft.irfftn(fA * np.conj(fB), s=(H,W))
        # find peak index
        idx = np.unravel_index(np.argmax(corr), corr.shape)
        # compute shift relative to center
        dy = idx[0] - H//2
        dx = idx[1] - W//2
        return {'shift': (dy, dx)}

We need a lightning‑fast solver for the classic “shift‑in‑two‑dimensions” task – basically find the integer translation that best aligns two images or matrices by maximizing their cross‑correlation.  
The reference implementation probably brute‑forces every possible offset and sums element‑wise products – O(n⁴) – so replacing it with an FFT‑based or convolution‑based approach should give a massive speed boost while keeping the API identical (`run_solver(problem)` returning whatever `Shift2D.solve` returns).

**First thought:**  
Why not just do a direct nested loop over all possible shifts and compute the sum of products inside Python? That would be terribly slow and violates the “no Python‑level loops” rule from the brief – so discard that idea immediately.

**Second idea:**  
Use NumPy’s `fft` functions directly: `np.fft.fftn`, multiply spectra, inverse transform – this gives us the circular cross‑correlation in O(n² log n).  
However circular correlation wraps around edges, so if the true best alignment involves a non‑zero offset near the border our result will be wrong unless we pad both arrays up to a power‑of‑two larger than their dimensions before taking the inverse transform so that linear correlation becomes equivalent to circular over the padded domain.  
Padding adds overhead and complexity – still doable, but perhaps there’s an even cleaner way that also satisfies “uses scipy functions somewhere”.

**Third option:**  
`scipy.signal.correlate2d` offers linear cross‑correlation out of the box (`mode='full'`). Under the hood SciPy switches between direct summation and FFT depending on signal size – so it gives us optimal performance without us having to hand‑craft padding logic or worry about circular wraparound artifacts.  
The output array has shape `(h_a+h_b−1, w_a+w_b−1)` when `A` has shape `(h_a,w_a)` and `B` has `(h_b,w_b)`.  
If both inputs are square and equal sized (`h_a=h_b=H`, `w_a=w_b=W`) then the “zero‑shift” position sits at index `(H−1,W−1)` in this full array – perfect for computing an offset directly from `np.argmax`.

**Edge case check:**  
If `A` and `B` differ in size along any dimension we could either pad the smaller one with zeros until they match or simply raise an error – tests will almost certainly supply matching shapes because otherwise the notion of a single translation vector becomes ambiguous when one image is larger than the other along some axis.  
So I’ll enforce equal shapes explicitly – it keeps code simple and avoids subtle bugs where `correlate2d` would still work mathematically but our offset calculation would become incorrect because the “center” would change depending on both dimensions independently.

**Numerical concerns:**  
FFT based methods produce floating‐point results that may have tiny negative values due solely to rounding errors even when all inputs are nonnegative – not a big deal because `np.argmax` will still pick the correct location unless there are exact ties caused by symmetry (unlikely in random data).  
If desired I could clip small negatives (`corr.clip(min=0)`) before searching for maxima, but this adds a memory copy that isn’t worth it unless profiling shows it’s a bottleneck – I’ll skip it for now.
