We need a fast `run_solver` that applies a 2‑D shift to an image without any Python‑level looping.  
The reference `Shift2D` class probably walks the pixels one by one – that’s slow on large arrays – so the obvious speed‑up is to delegate the heavy lifting to a library routine that already runs in compiled C/C++ code.  

The most natural candidate is `scipy.ndimage.shift`. It accepts a NumPy array and a `(dy,dx)` tuple of floats (sub‑pixel allowed), performs bilinear interpolation (`order=1`) in C, and returns an array of the same shape with boundary handling (`mode='nearest'` or `mode='constant'`).  
Using it keeps the implementation completely vectorised – no explicit Python loops are needed – while still giving us the sub‑pixel accuracy we usually want in a “shift” operation.  

Potential alternatives I toyed with:

* **FFT‑based phase ramp** – compute `np.fft.fftn`, multiply by `exp(-1j*2π*(k·shift)/N)` and inverse transform. Works for very large images when many shifts are needed, but the overhead of two FFTs outweighs the benefit for a single shift; plus we’d have to pad to avoid wrap‑around artefacts.
* **Numba JIT bilinear interpolation** – write a custom kernel that walks every pixel in compiled code. Still involves explicit indexing logic inside the JIT function – more complicated than calling an already battle‑tested routine.
* **OpenCV warpAffine** – would require importing `cv2` (not mentioned in the brief) and constructing an affine matrix even though we only need translation.
* **np.roll + local interpolation** – handles integer shifts cheaply but fails for fractional displacements unless you manually interpolate at the boundaries – essentially re‑implementing what `ndimage.shift` already does efficiently.
  
So I settled on `scipy.ndimage.shift`.  

Key design points:

1. **Imports**  
   ```python
   import logging
   from typing import Any, Dict
   import numpy as np
   import scipy.ndimage
   ```

2. **Logging** – create a module logger so that debugging information can be emitted without cluttering stdout during normal runs.
3. **Data handling**  
   * Pull the image from `problem['image']`.  
   * Accept an optional `'shift'` key (default `(0.,0.)`).  
   * Convert the image to `float64` before calling `shift` (the routine expects floats).  
   * After shifting cast back to the original dtype if possible (`np.can_cast`).
4. **Boundary mode** – use `'nearest'` so that pixels outside the original bounds are filled with the nearest edge value – this mimics many baseline implementations.
5. **No loops** – everything is a single function call into SciPy’s C code.
6. **Return format** – mimic what the baseline likely expects: just return the shifted array (or wrap it in a dict if required). For safety I’ll return the raw NumPy array because callers usually just want the data.
7. **Edge cases**  
   * Negative or huge shifts are handled automatically by `ndimage.shift`.  
   * If the input is already float32/float64 we preserve it after casting back.
8. **Performance** – The bulk of work happens inside SciPy’s BLAS/LAPACK routines; there’s no Python overhead beyond setting up arguments and casting types once per call.
9. **Type hints & docstrings** – minimal comments explaining why we chose this path without overloading the file size budget (~7500 characters).

With these pieces glued together into a small module containing only `run_solver`, we satisfy all constraints:

* Uses only NumPy & SciPy (plus standard library).  
* No Python loops anywhere in heavy computation paths.  
* Fully vectorised via SciPy’s internal implementation → significant speedup over naive pixel‑by‑pixel code.
* Keeps API identical (`run_solver(problem)` returning whatever `Shift2D().solve(problem)` used to return).
