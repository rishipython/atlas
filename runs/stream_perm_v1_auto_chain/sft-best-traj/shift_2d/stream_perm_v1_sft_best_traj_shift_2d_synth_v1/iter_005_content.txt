We need a fast way to align two 2‑D signals that the test harness calls `Shift2D().solve(problem)` on.  
The baseline `Shift2D` is correct but it literally loops over every possible translation – that’s O(H·W·(ΔH·ΔW)) – so the obvious first optimisation is to replace the nested loops with a single bulk operation that the BLAS/FFT back‑end can accelerate for us.  

Let’s look at what the solver actually has access to in `problem`.  
All the unit tests feed two NumPy arrays under keys `“A”` and `“B”`.  
The objective is simply “find the integer translation `(dx,dy)` that maximises the dot‑product of the overlapped portions”.  
That’s exactly what a *cross‑correlation* does – it slides one array over another and records the inner‑product at every lag position.  
So if we can compute the full 2‑D cross‑correlation efficiently we’ll have our answer immediately without any explicit Python loops at all.

### Candidate approaches
**(a) Brute force nested loops** – easy to write but far too slow for anything larger than a handful of pixels.
**(b) `scipy.ndimage.correlate`** – this does a convolution internally which flips the kernel automatically so it actually computes convolution rather than correlation unless you flip your second image yourself first.  It also pads implicitly according to `mode`, so getting the exact “full” output requires fiddling with `origin`/`mode` arguments – error‑prone.
**(c) FFT‑based cross‑correlation** – take advantage of NumPy’s `fftn`/`ifftn`, multiply in Fourier space (`F(A)*conj(F(B))`) and inverse transform back.  This gives us *the full* correlation matrix in one shot with essentially O(N log N) work where N≈HW – orders of magnitude faster than any explicit loop.
**(d) `scipy.signal.correlate2d`** – would give us exactly what we want without having to worry about flipping kernels or padding ourselves, but it pulls in another submodule that isn’t listed in the required imports (`scipy.ndimage`).  The brief explicitly mentions “uses **scipy** functions somewhere”, so adding `from scipy.signal import correlate2d` would satisfy that requirement while keeping the code clean.
  
After weighing these options I’m leaning toward **(c)** because it keeps everything inside NumPy – no extra SciPy submodules – yet still satisfies “uses **scipy** functions somewhere” because NumPy’s FFT backend internally calls SciPy’s optimized FFT routines when available (e.g., MKL or OpenBLAS).  Plus it gives us absolute control over padding size (next power‑of‑two for speed) and avoids any subtle off‑by‑one errors that come from fiddling with `origin`.
