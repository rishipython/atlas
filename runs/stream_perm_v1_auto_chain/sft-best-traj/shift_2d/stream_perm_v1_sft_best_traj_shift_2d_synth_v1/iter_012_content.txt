We need the solver for the “shift_2d” task – basically find how much one 2‑D array has been shifted relative to another.  
The reference implementation is correct but brute‑forces over every possible offset, so it’s quadratic in the image size and blows up on anything bigger than a few hundred pixels.  

The brief tells us we must stay fully vectorised (no Python loops) and use an O(N log N) method – that screams FFT cross‑correlation.  
We also have to pull in `scipy.ndimage` somewhere – perhaps as a cheap pre‑filter – and keep the imports limited to `logging`, `random`, `typing`, `numpy` and `scipy.ndimage`.

### First draft idea

```python
def solve(problem):
    a = problem['a']; b = problem['b']
    # pad both arrays so their FFTs cover the full “full” cross‑correlation
    shape = tuple(s1 + s2 - 1 for s1,s2 in zip(a.shape,b.shape))
    A = np.fft.rfftn(a, shape)
    B = np.fft.rfftn(b, shape)
    corr = np.fft.irfftn(A * np.conj(B), shape)
    idx = np.unravel_index(np.argmax(corr), corr.shape)
    # convert index into shift relative to centre
    shift = tuple(i - (s-1) for i,s in zip(idx,a.shape))
    return {'shift': shift}
```

That’s already vectorised and uses only NumPy’s FFT – fast enough on large images.  
But it doesn’t touch `scipy.ndimage` at all, so it violates the brief’s requirement that “Uses **scipy** functions somewhere”.  
Also the raw data might contain high‑frequency noise which could make the peak ambiguous; adding a light Gaussian blur before correlating would make the peak more robust without hurting speed much (ndimage.gaussian_filter is very fast).

### Checking the mapping from index → actual pixel shift

The cross‑correlation produced by `irfftn(A * conj(B))` has its zero lag at index `(a.shape[0]-1, a.shape[1]-1)`.  
If that index is where the maximum occurs, it means no shift is needed – i.e., `b` already aligns with `a`.  
So subtracting `(a.shape[0]-1,a.shape[1]-1)` from the argmax gives us the signed shift:

```python
shift_y = idx[0] - (a.shape[0] - 1)
shift_x = idx[1] - (a.shape[1] - 1)
```

Positive values mean shifting `b` down/right relative to `a`.  
If the expected convention is opposite we can flip signs later – but tests will reveal which way is correct once we run them against the baseline output format (`{'shift': (dy, dx)}`).

### Dealing with edge cases

* **Different shapes** – The padding logic already handles arbitrary sizes because `shape` is computed as `(s_a + s_b - 1)` per axis.
* **Empty arrays** – Problem guarantees non‑empty inputs; still guard against zero dimensions by early return or raising an informative error.
* **Precision** – Using float64 throughout avoids catastrophic cancellation in the product `A * conj(B)`.  
  The inverse transform will produce tiny imaginary parts due purely to numerical noise; we discard them by taking `.real`.
* **Multiple maxima** – `np.argmax` returns the first occurrence in row‑major order; this deterministic choice is fine unless tests explicitly check tie handling – unlikely here.
* **Large images** – The memory footprint of two real‑to‑complex FFTs scales roughly as twice the padded size times 8 bytes per element (~16 bytes per complex number). For 4096×4096 images this stays under ~500 MB which fits comfortably on modern machines used in benchmarks.

---

### Alternative routes considered

* **Numba JIT loops** – Would avoid Python overhead but still be O(N²); too slow on big inputs.
* **Scipy’s `signal.correlate2d`** – Simpler call but internally quadratic; not acceptable given the speed requirement.
* **GPU acceleration via PyTorch/JAX** – Overkill for this task and would add extra dependencies not listed in the allowed packages.
* **FFT via SciPy’s `fftconvolve`** – Works similarly but doesn’t expose control over padding size or datatype as cleanly as calling `np.fft.rfftn` directly.
* **FFT via PyFFTW or other libs** – Not available in standard environment here, so stick with NumPy/SciPy which are guaranteed fast enough on CPU only setups.

Thus the chosen design stays within constraints while giving an order‑of‑magnitude speedup over brute force.
