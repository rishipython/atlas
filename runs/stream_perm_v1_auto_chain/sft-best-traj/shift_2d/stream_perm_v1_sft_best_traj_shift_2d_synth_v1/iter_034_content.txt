We need to rewrite the slow Shift2D solver so that it uses fast SciPy routines and stays fully vectorised – no Python loops at all.  
The key operation in the original class is a 2‑D image shift (or batch of images).  
Scipy already has a highly optimised `ndimage.shift` that handles sub‑pixel offsets with spline interpolation in C code.  
So the plan is to replace whatever custom optimisation was happening inside `Shift2D.solve` with a single call to `scipy.ndimage.shift`.  

First question: what does the input look like? In the tests the `problem` dictionary always contains an `'x'` key holding either a single 2‑D array or a stack of them (shape `(N,H,W)`).  
There’s also a `'shift'` key giving the desired offset as a pair `(dy,dx)` – note that SciPy expects shifts in `(row,col)` order while the task usually supplies them as `(y,x)`.  
We’ll simply reverse the pair when passing it on.  

We also want to be defensive about shapes:

```
if x.ndim == 2:
    # single image
    out = scipy.ndimage.shift(x, shift=(dy,dx), order=1, mode='nearest')
elif x.ndim == 3:
    # batch of images – prepend a zero so all slices get the same offset
    out = scipy.ndimage.shift(x, shift=(0,dy,dx), order=1, mode='nearest')
else:
    raise ValueError(...)
```

`order=1` gives bilinear interpolation – fast enough and reasonably accurate; `mode='nearest'` keeps borders consistent with many baselines.  
No loops are needed because `ndimage.shift` internally processes the whole array at once (or all slices together when a leading dimension is present).

Logging is optional but handy for debugging – we’ll log the input shape and requested shift at DEBUG level so that any mis‑formatted input surfaces quickly without affecting speed in production runs where logging is disabled by default.

The function signature must match what the rest of the framework expects – `run_solver(problem)` returning whatever type the baseline returned.  
The reference `Shift2D.solve` returns just an array; therefore our wrapper will return the shifted array directly for compatibility.  

Edge cases to watch out for:

* The `'shift'` key might be missing – default to `(0.,0.)`.
* The value could be an integer or float; SciPy handles both.
* If someone passes a 4‑D tensor accidentally (e.g., `(B,N,H,W)`), our code will raise a clear error instead of silently misbehaving.
* Floating‑point rounding errors from spline interpolation are negligible compared to the tolerances used in evaluation (`rtol/atol`), so no clipping needed.
* For very large arrays SciPy’s internal FFT padding strategy automatically chooses efficient block sizes; no manual power‑of‑two padding required here.

With this design we keep everything vectorised, rely on SciPy’s C implementation for speed, avoid any Python‑level loops entirely, and stay within the requested imports (`logging`, `typing`, `numpy`, `scipy.ndimage`).
