We need to speed up the 2‑D shift routine so that it runs in \(O(N\log N)\) instead of the naïve \(O(N^2)\) approach that loops over every pixel and builds a big kernel or uses nested loops in Python. The problem statement gives us a `problem` dictionary that contains the input array and the desired shift vector – probably something like `{'data': np.ndarray, 'shift': (float, float)}` – and we must return the shifted array with the same shape and type as the input.

The key idea is to use the Fourier‑shift theorem: shifting an image by \((dx,dy)\) in real space corresponds to multiplying its Fourier transform by a complex exponential phase factor
\[
e^{-2\pi i \left(\frac{k_x\,dx}{N_x}+\frac{k_y\,dy}{N_y}\right)}.
\]
So we can:
1. Take the forward FFT of the data (`np.fft.fftn` or `scipy.fft.fftn`).
2. Build frequency grids `kx`, `ky` that match the shape of the FFT output using `np.arange` combined with `np.meshgrid`.  
   - We can get centered frequencies directly with `scipy.fft.fftfreq` which gives us values in \([-0.5,0.5)\); multiplying by \(N\) gives integer indices.
3. Compute the phase factor in one vectorised expression.
4. Multiply the FFT by this phase factor.
5. Inverse FFT back to real space (`np.ifftn`).
6. Take the real part (the imaginary part should be negligible) and cast back to the original dtype.
7. Return this result as the solution.

This approach has no Python‑level loops – everything is done in bulk on NumPy arrays – and it runs in \(O(N\log N)\) thanks to the underlying FFT implementation in NumPy/Scipy which calls highly optimised BLAS/LAPACK routines on modern CPUs (or even GPU if available).
