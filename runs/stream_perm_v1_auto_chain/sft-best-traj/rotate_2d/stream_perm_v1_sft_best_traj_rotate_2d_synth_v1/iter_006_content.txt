The task seems to take a two‑dimensional numeric array (an “image” or grid) and an angle in degrees and return the rotated grid. The reference implementation probably iterates over every pixel and applies a rotation matrix manually – that’s O(N²) in Python bytecode time and will kill us on large inputs.  

The brief tells us to import `scipy.ndimage`, so the obvious choice is `scipy.ndimage.rotate`. It’s written in C/Fortran, handles interpolation internally (bilinear by default), supports boundary modes (`constant`, `nearest`, etc.), and can keep the output shape fixed with `reshape=False`. All of this means we can drop any explicit loops entirely – just one function call does all the heavy lifting.  

We still have to parse the `problem` dictionary that comes in from AlgoTune’s harness. From past experience with these tasks I know that most problems expose their data under a key called `"data"` (or sometimes `"matrix"`). To stay robust we’ll look for both names and fall back gracefully if neither exists – raising a clear error would be better than silently returning garbage.  
The dictionary may also contain optional keys:
* `"angle"` – defaulting to `0`.
* `"reshape"` – defaulting to `False` so that the output keeps the same shape as the input (this matches many benchmarks).
* `"order"` – interpolation order (`1` for bilinear).
* `"mode"` – how out‑of‑bounds pixels are handled (`"constant"` by default).
* `"cval"` – constant value used when mode is `"constant"` (default `0`).

Because `ndimage.rotate` expects a NumPy array, we’ll cast whatever comes in (`list` or already‑array) into `np.asarray(..., dtype=np.float64)` – float64 gives us good precision without sacrificing speed on most machines.  
After rotation we convert back into a plain Python list-of-lists via `.tolist()` so that AlgoTune’s checker sees exactly what it expects (the reference solver does this too).

A few edge cases:
* Empty arrays (`arr.size == 0`) should return an empty list immediately – calling `rotate` on an empty array raises an exception.
* Angles that are multiples of 360° can be skipped entirely; this saves us a function call when the user asks for “no rotation”.
* Negative angles are handled automatically by `rotate`, so no special handling needed there.
* If the input has more than two dimensions (unlikely but possible), we’ll raise a ValueError because our solver only supports true 2‑D grids – this mirrors what most baseline implementations do when encountering unexpected shapes.
