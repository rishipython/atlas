Let’s sketch the core logic:

1. Extract image (`img`) from `problem['data']` or similar key.
2. Get rotation angle in degrees from `problem['angle']`.
3. Compute sine and cosine once.
4. Build a grid of coordinates covering the image using `np.indices`. This gives two arrays of shape `(H, W)` each.

6. Add back the centre offset after rotation.

8. Reshape back to `(H, W)` and return in whatever format the caller expects (likely just the rotated array).

Potential pitfalls:  
- The coordinate system of NumPy indices starts at 0 at the top‑left corner; rotating around the centre requires shifting by `(H/2 - 0.5, W/2 - 0.5)` rather than just `(H/2, W/2)` because pixel centres are offset by 0.5 in each direction.  
- Negative angles should work automatically because sine changes sign accordingly; no extra handling needed.  
- For very small images or very large angles close to multiples of 360°, rounding errors could cause tiny negative values after interpolation; but since we’re using linear interpolation on integer arrays this should be fine.

We also want to keep everything vectorised: no explicit Python loops over pixels or rows; all heavy lifting is done by NumPy/SciPy calls that internally call BLAS/LAPACK routines for large matrix multiplications or convolution kernels.
