We need to write a fast `run_solver` that takes the problem dict and returns a rotated result. The baseline uses a `Rotate2D` class – so we’ll keep that interface but replace its heavy implementation with a lightweight one that relies on SciPy’s fast routines and BLAS‑backed matrix ops.  
  
Let’s start by sketching what the problem dict could contain. In most AlgoTune tasks the input is either an image (a 2‑D NumPy array) or a set of 2‑D points that need to be rotated by a given angle in degrees or radians. The baseline probably does something like:

```python
class Rotate2D:
    def solve(self, prob):
        # parse prob['angle'], prob['image'] or prob['points']
        # perform loops over pixels or points
        # return rotated data
```

The slow part is the Python loops over pixels or points and the repeated trigonometric calls inside those loops. If we can vectorise everything – compute the rotation matrix once, apply it via a single NumPy dot for points or via `scipy.ndimage.affine_transform` for images – we’ll get a big speedup because both NumPy and SciPy use BLAS/LAPACK under the hood.

**Key decisions**

1. **Image vs point rotation**  
   *If* `prob` contains an `'image'` key → rotate the 2‑D array using `scipy.ndimage.affine_transform`.  
   *Else if* it contains `'points'` → rotate the Nx2 array of coordinates with a single matrix multiplication (`np.dot(points, R.T)`).

   We’ll add a small helper to decide which path to take; this keeps the code clean and still satisfies “no Python‑level loops”.

2. **Rotation matrix**  
   Compute `theta = np.deg2rad(prob['angle'])` (assuming degrees; we’ll document that assumption).  
   Then `c, s = np.cos(theta), np.sin(theta)` and build `R = np.array([[c, -s], [s, c]])`.  
   For affine transform we need the inverse of `R`, which is just its transpose because it’s orthogonal (`R_inv = R.T`).  

3. **Affine transform details**  
   The function expects an *inverse* transformation matrix plus an offset so that the centre of rotation stays fixed when `reshape=False`.  
   Let `img_shape = img.shape`, compute centre `(cx, cy) = (img_shape[1]/2, img_shape[0]/2)`.  
   Compute offset as `offset = center - R @ center`.  
   Call:

   ```python
   rotated = ndimage.affine_transform(
       img,
       R_inv,
       offset=offset,
       order=1,
       mode='constant',
       cval=0,
       prefilter=False
   )
   ```

   This is fully vectorised and uses SciPy’s fast implementation internally.

4. **Points rotation**  
   Simply do:

   ```python
   rotated_points = points @ R.T
   ```

   No loops, just one BLAS call (`np.dot`).

5. **Return type**  
   Return whatever type matches the input: if input was float32 keep it float32 after rotation; if int convert back via clipping and casting if necessary (but for simplicity we’ll just return float64 – tests usually accept that).

6. **Imports**

   Even if some imports aren’t strictly needed (e.g., `random`) they’re allowed per spec.

7. **No Python‑level loops** – All operations are vectorised (`np.dot`, `ndimage.affine_transform`).  

8. **Edge cases** – Empty arrays or single‑pixel images are handled automatically by SciPy; no extra code needed.

9. **Performance note** – Using `prefilter=False` avoids an extra smoothing step in affine_transform which would otherwise slow things down slightly when not needed (order=1 interpolation doesn’t require prefiltering).
