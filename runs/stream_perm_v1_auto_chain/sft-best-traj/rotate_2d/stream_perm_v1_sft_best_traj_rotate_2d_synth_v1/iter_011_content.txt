We need to take the `problem` dict that contains a set of 2‑D coordinates and an angle and output the coordinates after rotating them around the origin by that angle. The reference implementation does this point‑by‑point in pure Python – slow because it has a Python loop over every point and builds the rotation matrix on each iteration. We want to replace that with a fully vectorised approach that still calls into SciPy so we satisfy the “uses SciPy” requirement while keeping all heavy lifting in compiled code (BLAS/LAPACK).

Let’s think about the maths first. A 2‑D rotation by an angle θ can be written with the standard rotation matrix

```
R(θ) = [[cos θ , -sin θ],
        [sin θ ,  cos θ]]
```

If we simply construct this matrix ourselves with `np.cos`/`np.sin` and then do a single matrix multiplication `points @ R.T`, everything is vectorised and runs on BLAS – very fast even for millions of points. However that would mean we never actually touch SciPy in our own code path – the brief explicitly wants us to call a SciPy routine somewhere in the solution flow. One simple way is to generate `R` via the matrix exponential of a skew‑symmetric generator:

```
K = [[0 , -1],
     [1 ,  0]] * θ
R = expm(K)
```

`scipy.linalg.expm` does exactly that and internally uses LAPACK routines – so it satisfies the SciPy usage clause while keeping our own code free of loops and still very fast because `expm` on a fixed 2×2 matrix is essentially constant work regardless of how many points we later multiply by it.

So the plan is:

1. Pull out `points` and `angle` from `problem`. Convert `points` into a NumPy array (`float64`) – this automatically handles lists/tuples/ndarrays coming in from the test harness.
2. Build the generator `K` by multiplying the fixed skew‑symmetric template by `angle`. No branching here.
3. Call `scipy.linalg.expm(K)` to get `R`. This is our one explicit SciPy call.
4. Multiply all points at once with `R.T` (`pts @ R.T`). NumPy will dispatch this to BLAS when available – satisfying “matrix multiplication (BLAS‑backed)” requirement.
5. Return the result as a plain Python list (`tolist()`). The reference implementation probably expects a serialisable structure rather than a NumPy array itself, so converting back keeps compatibility without adding any extra overhead beyond what NumPy already pays for during conversion anyway.
6. Edge cases are trivial:
   * If there are zero points (`pts.shape[0] == 0`) the dot product yields an empty array which converts cleanly to an empty list.
   * A zero angle gives `K=0`, so `expm(0)` returns identity – no change.
   * Negative angles are handled automatically because `K` gets sign flipped accordingly.
7. No Python loops appear anywhere in our own logic – all heavy work stays inside NumPy/SciPy compiled kernels.
8. Import style matches brief (`from __future__ import annotations`, typing imports).

That’s basically it – just one line of math plus one SciPy call plus one BLAS‑backed dot product gives us a massive speedup over per‑point looping while staying within all constraints given in the brief.
