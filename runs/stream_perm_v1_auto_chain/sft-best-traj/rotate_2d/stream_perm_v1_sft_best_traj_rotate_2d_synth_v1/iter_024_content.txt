We need to replace the slow baseline Rotate2D implementation with something that uses SciPy’s fast routines and stays fully vectorised – no Python loops at all.  
Let’s start by figuring out what the input looks like. The `solve` method in the reference class probably receives a dictionary with an image array and an angle (in degrees). The tests will feed it via `run_solver(problem)` where `problem` is a plain dict, so we just unpack it there.

The obvious candidate for fast rotation is `scipy.ndimage.rotate`. It already handles reshaping, interpolation orders, and works on NumPy arrays directly. If we can just hand off the data to that function we’ll get a huge speedup compared to a manual loop over pixels or a custom bilinear interpolation routine.

**What arguments do we need?**  
- `input_array`: whatever shape the problem gives us – likely 2‑D but could be multi‑channel (e.g., RGB).  
- `angle`: float degrees – `ndimage.rotate` expects degrees by default so no conversion needed unless the baseline used radians (we’ll check the tests).  
- `reshape`: bool – keep it true by default but allow override via problem dict (`'reshape': False`).  
- `order`: interpolation order – the baseline probably used bilinear (order=1) for speed vs accuracy trade‑off; cubic (order=3) gives nicer results but slower. We’ll expose it as optional too (`'order'`).  

**Edge cases**  
- If the image is 1‑D or empty – return it unchanged (ndimage.rotate would error).  
- Preserve dtype: if input is float32 we should keep that; ndimage will cast internally but we can cast back with `.astype(img.dtype)`.  
- If angle is NaN or infinite – just return the original array (the baseline probably handled this gracefully).  

**Implementation sketch**  
```
def run_solver(problem):
    img = np.asarray(problem['image'])
    if img.ndim < 2:
        return img
    angle = float(problem.get('angle', 0))
    reshape = bool(problem.get('reshape', True))
    order = int(problem.get('order', 1))
    rotated = scipy.ndimage.rotate(img, angle, reshape=reshape,
                                   order=order, mode='constant', cval=0)
    return rotated.astype(img.dtype)
```
That’s all vectorised – one call to SciPy and a couple of lightweight NumPy ops.

Now think about potential pitfalls:  
- **Performance**: `ndimage.rotate` internally uses FFT for large images when `reshape=True`; that’s fine and fast enough for our purposes.  
- **Memory**: For very large images the intermediate buffers might be big; but that’s unavoidable when rotating with reshape because you need a bigger output canvas.  
- **Interpolation mode**: Using `'constant'` with `cval=0` matches many test expectations (background zero). If tests expect edge replication we’d switch to `'reflect'`. We’ll stick with constant because it’s fastest and most common in benchmarks.  

Alternative designs considered:  
1. **Numba JIT** over a nested loop doing bilinear interpolation manually – would give decent speed but still involves Python-level loops at compile time and more code complexity than necessary; plus SciPy already has highly optimised C code for this task.
2. **Custom FFT-based rotation** using polar coordinates – overkill for this problem size and would add numerical instability.
3. **Using OpenCV (`cv2.warpAffine`)** – not in allowed packages list unless installed; also adds dependency overhead.
