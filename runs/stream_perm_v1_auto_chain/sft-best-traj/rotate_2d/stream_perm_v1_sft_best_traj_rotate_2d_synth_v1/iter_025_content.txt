Let’s think about the minimal set of operations required:

```
input : ndarray `data`   # could be H×W or N×H×W or even C×H×W after a small shuffle
angle : float            # degrees – the usual convention used by SciPy’s helpers
output : ndarray          # same shape & dtype as `data`
```

The SciPy API that does exactly what we want is `scipy.ndimage.affine_transform`.  
It takes an arbitrary linear transform matrix **M** that maps *output* coordinates back into *input* coordinates,
an optional `offset` vector that compensates for the fact that rotations are usually defined about the origin,
and it works on any number of dimensions – the transform is applied independently along each axis unless you explicitly mix them in **M**.
The heavy lifting happens in C/Fortran so there are no Python loops involved once you hand it the right arguments.



### Building the transform

For a pure spatial rotation by an angle θ around the centre of an H×W image:

```
M₂ₓ₂ = [[cosθ , -sinθ],
        [sinθ ,  cosθ]]
```

The centre in pixel coordinates is  
`(cy , cx) = ((H‑1)/2 , (W‑1)/2)` .  
If you simply apply `M₂ₓ₂` you’ll end up rotating around the origin `(0,0)` which moves the centre away.
To keep the centre fixed you add an `offset` computed from

```
offset = centre – M₂ₓ₂ @ centre
```

When `data` has more than two spatial dimensions – e.g., a stack `N×H×W` or a colour image `C×H×W` –  
we just prepend an identity row/column so that the extra dimension is untouched:

```
M₃ₓ₃ = [[1 , 0 ,   0],
         [0 , cosθ , -sinθ],
         [0 , sinθ ,  cosθ]]
offset₃ₓ₁ = [0 , offset_y , offset_x]
```

This way every “slice” along the leading axis gets rotated identically without any explicit looping.



### Handling colour / batch inputs

A common pitfall is colour images stored as `(H,W,C)` while our transform expects the spatial axes last.
A quick `np.moveaxis(data,-1,0)` turns it into `(C,H,W)` – now **C** behaves like an extra leading dimension,
exactly what `affine_transform` already knows how to ignore thanks to the identity part of **M**.
After rotation we move it back with another `moveaxis`.

If somebody hands us something already shaped `(N,H,W)` there’s nothing special needed –
the same logic applies.
