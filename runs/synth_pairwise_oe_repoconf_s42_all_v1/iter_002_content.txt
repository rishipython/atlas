**Option A – pure NumPy**  
```

```
This is already blazing fast on modern CPUs because `@` dispatches to BLAS automatically.  
But it doesn't touch `scipy`, nor does it involve any `numba`.  
So while it's fast enough, it violates two of the constraints – not acceptable.

**Option B – use `scipy.spatial.distance.cdist`**  
```

```
That’s a single call and probably pretty fast too because SciPy implements it in C/C++.  
However `cdist` internally uses its own algorithm – likely not purely BLAS‑based – and I still haven’t used `numba`.  
Also I’d have to square the result again or tweak parameters – adds unnecessary gymnastics.  
Thus I’ll drop this idea.

**Option C – explicit BLAS via `scipy.linalg.blas`**  
SciPy exposes low‑level BLAS wrappers (`sgemm`, `dgemm`, etc.).  
If I call `dgemm(alpha=1., a=X, b=X.T)` I get exactly the same cross‑product matrix as `np.dot`, but with an explicit reference to SciPy’s BLAS layer – satisfying that requirement neatly.

Now I’ve used SciPy explicitly and still rely on BLAS for speed.
