We need a fast way to get the full pairwise squared distance matrix without looping over every pair in Python – that’s what kills the reference code’s speed when N grows beyond a few hundred points.

The classic trick is  
\(D_{ij}=\|x_i\|^2+\|x_j\|^2-2\,x_i\!\cdot\!x_j\).  
If we pre‑compute the column‑wise sums of squares \(s_i=\sum_k x_{ik}^2\) and the Gram matrix \(G_{ij}=x_i\!\cdot\!x_j\), then the whole distance matrix is just  
`s[:,None] + s[None,:] - 2*G`.  
The heavy lifting – the Gram product – can be delegated straight to BLAS through SciPy’s low‑level interface (`scipy.linalg.blas`).  
That gives us a single massive matrix multiply backed by highly optimised C/Fortran code – far faster than any pure‑Python approach.

We still have two remaining constraints from the brief:
* **Use `numba`** – so let’s wrap the per‑row norm calculation in a tiny JIT‑compiled routine that runs in parallel over rows.
* **No Python‑level loops** – everything else stays vectorised or handled by the underlying libraries (`numpy`, SciPy).

Edge cases are naturally handled:
* If `N==0`, the Gram product returns an empty `(0,0)` array and broadcasting produces an empty result – correct.
* If `D==0`, every row norm is zero and the Gram product is all zeros – again giving an all‑zero distance matrix.
* The formula never introduces negative values except possibly through catastrophic cancellation when two identical points are represented with floating‑point noise – this would only produce tiny negatives well within any reasonable tolerance (`rtol/atol`) used by the tests.
